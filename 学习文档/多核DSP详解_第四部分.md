## 第 9 章 开发工具链

### 9.1 工欲善其事，必先利其器

写 C6678 代码不像写 Arduino 或 STM32 那样有成百上千个开源 IDE 和工具链可选。TI 的生态是一套**相对封闭但完整**的工具链，主要就围绕一个 IDE：**CCS（Code Composer Studio）**。本章走马观花介绍开发工具链里的关键组成部分，让你对"这些东西是干什么用的"有个概念，遇到具体问题时知道去哪里查。

### 9.2 CCS：TI 的官方 IDE

**CCS** 的全称是 **Code Composer Studio**，是 TI 为自家芯片提供的集成开发环境。它基于 **Eclipse** 框架做了大量定制，所以如果你用过 Eclipse 系列 IDE，上手会很快。

CCS 提供的主要功能：

- **工程管理**：创建、编辑、编译 DSP 工程
- **C/C++ 代码编辑器**：带语法高亮、自动补全、重构
- **编译器和链接器集成**：TI 的 C6000 编译器（ti-cgt）
- **调试器**：单步、断点、内存查看、寄存器查看
- **多核调试**：一个窗口同时调试 8 个核（重点功能）
- **目标配置**：告诉 CCS 你的板子用的是什么仿真器、什么芯片
- **烧录工具**：把 `.out` 文件烧到 SPI Flash/NAND 里

在本项目里，你会看到每个核的工程目录下都有 `.project`、`.cproject`、`targetConfigs/TMS320C6678.ccxml` 这些文件——这些都是 CCS 的工程配置文件。其中 `.ccxml` 特别重要，它定义了"**用哪个仿真器连哪块板子**"。从 `MCB_CORE0/targetConfigs/TMS320C6678.ccxml` 可以看到本项目用的是 **Blackhawk XDS560v2-USB** 这款高速仿真器：

```xml
<instance desc="Blackhawk XDS560v2-USB System Trace Emulator_0" .../>
<device description="C66x core" id="TMS320C6678_0" partnum="TMS320C6678">
    <cpu description="C66xx CPU" id="C66xx_0" isa="TMS320c66xx">
        <property Value="..\gel\DSP_C6678.gel" id="GEL File"/>
    </cpu>
</device>
```

**XDS560v2** 是 TI 和 Blackhawk 合作的高性能 JTAG 仿真器，支持 C6678 的系统跟踪（System Trace）——能记录运行时的事件序列，对多核调试非常有用。普通项目用更便宜的 **XDS100/XDS200** 也能做基础调试，只是不支持高速跟踪。

**GEL 文件**（`DSP_C6678.gel`）是 CCS 的"初始化脚本"。GEL 的全称是 **General Extension Language**（通用扩展语言），是一种类似 C 的脚本，在调试器连上芯片时自动执行——比如"给 DDR3 控制器写初始化参数让 DDR3 能用"、"设置 PLL 到 1 GHz"。没有 GEL 的话，刚上电的 C6678 处于默认状态，很多外设都没初始化，调试器直接跳进代码会碰到各种问题。

### 9.3 编译器：TI C6000 CGT

TI 提供的 C6000 编译器叫 **CGT**（Code Generation Tools，代码生成工具），通常版本号形如 `C6000_7.4`。从 `.cproject` 文件里可以看到项目用的是这个版本。

CGT 包含：

- **cl6x**：C/C++ 编译器
- **asm6x**：汇编器
- **lnk6x**：链接器
- **ar6x**：静态库归档工具
- **hex6x**：生成烧录用的十六进制文件

**C6000 编译器是整个 DSP 性能的灵魂**。前面第 3 章讲过，VLIW 架构的指令打包**由编译器决定**，编译器优化水平直接决定最终性能。C6000 编译器在几个方面做了大量工作：

1. **软件流水**（Software Pipelining）：把循环体展开、重排，让多个迭代在不同流水阶段并行执行
2. **自动向量化**：识别循环里的并行机会，生成 SIMD 指令
3. **VLIW 打包**：把独立指令尽可能塞到同一个指令包里
4. **寄存器分配**：把热点变量放在寄存器里
5. **指令调度**：重排指令顺序以减少流水线停顿

这些优化靠编译选项开启：

- **`-O0`**：不优化，代码最容易调试，性能最差
- **`-O1`**：基本优化
- **`-O2`**：更多优化，生产代码的常用级别
- **`-O3`**：最高优化，包含激进的内联、循环展开等
- **`-o3`**（小写）：等同于 `-O3`，TI 的历史写法

**开发过程中的典型做法**：调试时用 `-O0`（代码和源码一一对应，断点好用），发布时用 `-O3`（最大性能）。但要注意——`-O3` 下编译器会做很多重排，某些看似正常的代码可能被"优化掉"（比如编译器判断某个变量没用就直接删了），这时候要配合 `volatile` 关键字告诉编译器"这个变量别乱动"。

### 9.4 SYS/BIOS、TI-RTOS 还是裸机

写 DSP 代码可以带操作系统也可以不带。选择有三种：

#### 9.4.1 裸机（Bare Metal）

**没有操作系统**，程序从 `main()` 开始一直跑到关机，所有调度靠中断和状态机。

**优点**：
- 最小开销，最短延迟
- 代码结构简单，行为可预测
- 可认证性最好

**缺点**：
- 多任务管理全靠手动写状态机
- 没有任务优先级概念
- 大型项目容易代码混乱

**适用场景**：硬实时控制、安全关键系统。

#### 9.4.2 SYS/BIOS（现在叫 TI-RTOS）

TI 自家的**轻量级实时操作系统**，最早叫 DSP/BIOS，后来改名 SYS/BIOS，现在统一到 TI-RTOS 品牌。特点：

- **轻量**：内核只有几十 KB
- **实时**：硬实时调度，抢占式优先级
- **集成**：和 CCS 深度集成，可以图形化配置（通过 `.cfg` 文件）
- **免费**：TI 随 CCS 一起提供

**提供的能力**：
- 多任务（Task）
- 硬件中断（Hwi）和软件中断（Swi）
- 信号量、邮箱、队列
- 时钟和定时器
- 内存管理

**适用场景**：任务数量多、需要模块化的中等复杂度项目。

#### 9.4.3 本项目的选择：偏向裸机 + 中断驱动

从代码结构看，本项目的 CORE0 使用的是一种**偏裸机的事件驱动模型**，没有明显的任务调度器痕迹：

- 核心入口是 Timer 中断触发的 `IsrMotionControl()`
- 中断服务函数里用状态机（switch-case）管理不同的系统状态
- 没有看到 `Task_create()`、`Semaphore_pend()` 这类 SYS/BIOS API
- 以太网部分用了 LWIP，但 LWIP 是协议栈而不是 OS

这种"**裸机 + 中断驱动 + 状态机**"的模式在运动控制里非常典型。整个系统的行为由硬件定时器和硬件中断驱动，代码路径高度确定，没有调度器带来的不确定性。代价是代码组织完全依赖程序员的经验，大型项目容易变成一堆互相交错的状态机。

### 9.5 多核调试：CCS 的杀手级功能

单核调试相对简单，但**多核调试**是另一个级别的挑战——你有 3 个核同时在跑，要同时断点、同时观察、同时单步。CCS 对多核调试的支持是 TI 的一大卖点。

#### 9.5.1 多核调试会话

在 CCS 里连接上 C6678 之后，调试窗口会列出 **8 个核**作为独立的调试目标（即使你只用了其中几个）：

```
Debug
├── Blackhawk XDS560v2-USB / C66xx_0 [CORE0]
├── Blackhawk XDS560v2-USB / C66xx_1 [CORE1]
├── Blackhawk XDS560v2-USB / C66xx_2 [CORE2]
├── ...
└── Blackhawk XDS560v2-USB / C66xx_7 [CORE7]
```

你可以**给每个核加载不同的 `.out` 文件**——这就是 AMP 的调试方式。CORE0 加载 `MCB_CORE0.out`，CORE1 加载 `MCB_CORE1.out`，各跑各的代码。

#### 9.5.2 同步运行和同步断点

CCS 提供 **Group** 功能：把多个核放到一个调试组里，然后"**组运行**"和"**组暂停**"——也就是 8 个核同时开始执行，同时被拦截。这在调试"核间协作"的 bug 时特别重要，比如你想观察 CORE0 发 IPC 中断后 CORE1 响应的完整时序，就需要两个核同时停在各自的相应位置上。

还有**同步断点**：你在 CORE0 的某一行设断点，同时在 CORE1 的某一行设断点，任何一个断点触发都会让**整个组都停下来**。这样你就能在两个核的不同位置"拍照"——分别看各自的寄存器、内存、变量状态。

#### 9.5.3 System Trace 和 CTools

高级调试功能还有：

- **System Trace（系统跟踪）**：硬件记录运行时事件（函数调用、中断、指令流），可以回溯分析"bug 发生前的 1000 条指令是什么"
- **CTools**：CCS 里的统计分析工具，能算出函数执行时间、Cache 命中率、指令混合度等
- **ROV（Runtime Object View）**：TI-RTOS 的运行时对象查看器，能看到所有任务、信号量、队列的状态

这些高级功能不是每天都用，但遇到棘手问题时是救命稻草。

### 9.6 烧录与启动流程

写完代码调试完了，最后要把程序**永久烧到板子上**，让板子上电就自动运行。这个过程叫**烧录**（flashing）或**固化**。

#### 9.6.1 C6678 的启动方式

C6678 支持多种启动方式，由板子上的 **BOOTMODE** 引脚决定上电时从哪里读程序：

- **从 SPI Flash 启动**：最常见，SPI Flash 里存放压缩过的启动镜像
- **从 NAND Flash 启动**：适合程序体积大的场景
- **从 I2C EEPROM 启动**：小规模启动代码
- **从以太网启动**（PCIe Boot、Ethernet Boot）：开发调试时用
- **从 SRIO 启动**：由另一颗芯片（FPGA）提供启动镜像

本项目同时有 `spiflash/` 和 `nand/` 目录，但通常实际启动介质是 SPI Flash——因为 SPI Flash 接口简单、启动速度快、可靠性高。NAND 可能用来存储大容量的配置数据或日志。

#### 9.6.2 两阶段启动（Two-Stage Boot）

C6678 在复杂系统里通常用**两阶段启动**：

**第一阶段：ROM Bootloader**

上电时 C6678 芯片内部有一个固化的 **ROM Bootloader**（自带的，无法修改），它会：

1. 读 BOOTMODE 引脚，判断启动源
2. 如果是 SPI Flash，就通过 SPI 读取前几 KB 作为**二级引导程序**
3. 把二级引导程序加载到 L2 SRAM，跳转过去执行

**第二阶段：Secondary Bootloader（SBL）**

二级引导程序通常由开发者自己写或用 TI 提供的 IBL（Intermediate Bootloader）。它负责：

1. 初始化 DDR3 控制器
2. 配置 PLL 到目标频率（1 GHz）
3. 初始化其他必要的外设
4. 从 SPI Flash 或 NAND 读取**真正的应用程序镜像**
5. 把镜像加载到各核的 L2 或共享内存
6. 释放各从核的复位信号
7. 跳转到 CORE0 的 `_c_int00`（C 运行时入口）

**为什么要两阶段？** 因为 ROM Bootloader 能力有限（只能读前几 KB），而实际应用程序可能几百 KB 甚至几 MB。先用 ROM 加载一个小的二级引导程序，再由二级程序去加载大的应用程序，就能突破这个限制。

#### 9.6.3 多核镜像的组织

AMP 模式下有多个 `.out` 文件（CORE0/CORE1/CORE7 各一个），它们怎么一起烧到 SPI Flash？通常的做法是：

1. 用 TI 提供的 `hex6x` 工具把每个 `.out` 转成**二进制镜像**
2. 用 **MulticoreImageGen** 工具把多个核的二进制镜像合并成一个大的**多核启动镜像**
3. 给这个镜像加上启动头（指定每个核的入口地址、加载地址）
4. 烧到 SPI Flash

启动时，SBL 读出这个大镜像，按头里的信息把各个核的部分分别加载到各自的 L2，然后启动它们。

### 9.7 第 9 章小结

1. **CCS** 是 TI 的官方 IDE，基于 Eclipse，支持多核调试
2. **XDS560v2** 等仿真器通过 JTAG 连接 C6678，提供调试和跟踪
3. **GEL 文件**在调试器连上芯片时自动执行初始化脚本
4. **TI C6000 编译器**的优化级别直接影响性能，`-O3` 是生产代码的常用选项
5. **裸机 + 中断驱动 + 状态机** 是运动控制项目的典型风格（本项目采用）
6. **多核调试**通过 CCS 的 Group 功能实现同步运行和同步断点
7. **两阶段启动**：ROM Bootloader → SBL → 应用程序，多核镜像用 MulticoreImageGen 合并烧录

## 第 10 章 性能优化要点

### 10.1 什么时候需要优化

有句话叫"**过早优化是万恶之源**"。在 DSP 开发里，这句话需要修正：**架构决策阶段的"优化"是必须的，局部代码的"优化"往往是多余的**。

架构决策指的是——

- 数据放 L2 还是 MSMC 还是 DDR3
- 用不用 EDMA 代替 memcpy
- 核间通信是裸共享内存还是 IPC 中断
- Cache 一致性怎么处理

这些决定会影响整个项目的性能量级，必须在项目早期就想清楚，否则后期改动代价极大。

局部优化指的是——

- 手写汇编
- 循环展开
- 调用内在函数
- 手动向量化

这些优化只有在**实测发现某段代码是瓶颈**之后才去做。盲目优化不仅浪费时间，还会让代码变得难以维护。

**优化的正确流程**：

1. 用 CCS 的 Profiler 或手工埋点测量各段代码的执行时间
2. 找出真正的瓶颈（通常 80% 的时间花在 20% 的代码上）
3. 针对瓶颈做优化
4. 再测，确认优化有效
5. 重复

本章介绍的优化技巧都应该在上述流程的第 3 步才用得上。

### 10.2 EDMA 双缓冲（Ping-Pong Buffer）

这是 DSP 优化里**最经典、收益最大**的一个技巧，每个 DSP 工程师都应该掌握。

#### 10.2.1 问题场景

假设你的 CORE0 要从 FPGA 不停地读一批传感器数据做处理，朴素的做法是：

```
while (true) {
    EDMA_Transfer(FPGA -> buffer);   // 启动搬运
    Wait_EDMA_Done();                // 等搬完
    Process(buffer);                  // 处理数据
}
```

**这个做法的问题**：EDMA 搬运和 CPU 处理是**串行**的。EDMA 在搬的时候 CPU 闲着，CPU 在处理的时候 EDMA 闲着。两个硬件资源都没充分利用。

#### 10.2.2 双缓冲的思路

用**两个缓冲区** A 和 B，让 EDMA 和 CPU **同时工作**——EDMA 搬运到 A 的时候，CPU 在处理 B 里上一轮的数据；下一轮交换角色。

```
轮次 1:  EDMA 搬到 A   |  CPU 闲（第一轮没数据）
轮次 2:  EDMA 搬到 B   |  CPU 处理 A
轮次 3:  EDMA 搬到 A   |  CPU 处理 B
轮次 4:  EDMA 搬到 B   |  CPU 处理 A
...
```

代码大致长这样：

```c
uint8_t buffer[2][DATA_SIZE];
int ping = 0;

// 启动第一轮 EDMA 到 buffer[0]
EDMA_StartTransfer(FPGA, buffer[0], DATA_SIZE);

while (true) {
    int pong = 1 - ping;

    // 启动下一轮 EDMA 到 buffer[pong]
    EDMA_StartTransfer(FPGA, buffer[pong], DATA_SIZE);

    // 同时处理上一轮的 buffer[ping]
    Process(buffer[ping]);

    // 等这一轮 EDMA 完成
    Wait_EDMA_Done();

    ping = pong;
}
```

这种技术叫**双缓冲**或**乒乓缓冲**（Ping-Pong Buffer）。理论上它能把吞吐率提高一倍（在 EDMA 时间和 CPU 处理时间相当的理想情况下）。

#### 10.2.3 运动控制里的应用

本项目的 `IsrMotionControl()` 里其实就用了类似思想：

```c
ReadSensorTopHalf();        // 启动 EDMA（上半部分，只启动不等待）
ReceiveControlCommand();    // 同时 CPU 去做别的事
CheckSensorDataTransfer(40); // 回头等 EDMA 完成
ADDataProcess();
```

这里 `ReadSensorTopHalf()` 只是**启动** EDMA，然后马上返回，CPU 接着去做 `ReceiveControlCommand()`（和上位机通信的处理），等做完了再回来检查 EDMA 是不是搬完了。这是"**CPU 和 EDMA 并行工作**"的经典体现。虽然不是严格意义上的"两个缓冲区轮换"，但思想是一样的：**让 EDMA 和 CPU 同时干活**。

### 10.3 Cache 优化与数据摆放

#### 10.3.1 Cache 的基本性能特征

前面讲过，L1D Cache 命中的访问是 1 周期，缺失要几十到几百周期。所以 Cache 命中率对性能影响巨大。提高命中率的几个原则：

**原则 1：空间局部性**——访问一个地址时，很可能也要访问附近的地址。把**相关数据放在一起**能提高命中率。

反例：

```c
struct BadLayout {
    float x;
    float unrelated_stuff[100];    // 中间夹了一堆不相关的
    float y;
};

// 访问 x 和 y 时要跨越一大段内存
```

正例：

```c
struct GoodLayout {
    float x;
    float y;    // 紧挨着 x，可能在同一个 Cache 行里
    // ...其他不相关的放后面
    float unrelated_stuff[100];
};
```

**原则 2：时间局部性**——刚用过的数据很可能很快再用。**循环里尽量复用已加载到 Cache 的数据**。

反例（列遍历一个大数组）：

```c
for (int col = 0; col < N; col++) {
    for (int row = 0; row < M; row++) {
        sum += matrix[row][col];   // 每次跳 N*sizeof(float) 字节
    }
}
```

这种访问方式每次跳一整行，Cache 行加载了一整行的数据但只用了一个元素，浪费严重。

正例（行遍历）：

```c
for (int row = 0; row < M; row++) {
    for (int col = 0; col < N; col++) {
        sum += matrix[row][col];   // 连续地址，Cache 行利用率高
    }
}
```

一字之差，性能可能差几十倍。

#### 10.3.2 结构体对齐与填充

前面讲 Cache 一致性时提到的**伪共享**问题也是 Cache 优化的一部分。为了让每个共享变量独占 Cache 行，要用对齐：

```c
typedef struct {
    volatile int state;
    char padding[124];    // 填充到 128 字节
} __attribute__((aligned(128))) AlignedSlot;

AlignedSlot slots[8];
```

这样 `slots[0]` 到 `slots[7]` 每个都在独立 Cache 行里，多核并发写不同 slot 不会互相干扰。

#### 10.3.3 关键数据放 L2

第 4 章讲过，本项目的链接脚本把所有段都放 L2 SRAM，避免了 Cache Miss。如果你的项目涉及 DDR3 里的大数据，而里面只有一小块热点数据，可以考虑：

1. 用 EDMA 把这块热点数据**从 DDR3 搬到 L2** 做处理
2. 处理完再搬回 DDR3
3. 期间所有访问都在 L2，速度极快

这个模式是"**DSP 的运算循环都在 L2 里**"，是 TI 推荐的经典做法。

### 10.4 内在函数（Intrinsics）

#### 10.4.1 什么是 Intrinsics

前面第 3 章说过，C66x 的 VLIW/SIMD 能力要靠编译器发挥，但编译器不是万能的——有时候你写的 C 代码编译器分析不出来能并行，或者你想用某个特定的 DSP 指令。这时候可以用**内在函数**（Intrinsics）。

**Intrinsics 是"看起来像 C 函数，实际编译成特定汇编指令"的东西**。你在 C 代码里直接调用它们，编译器会把它们展开成对应的汇编指令，绕过"编译器能否识别"这一步。

TI C6000 编译器提供了**几百个 Intrinsics**，对应 C66x 的各种特殊指令。常用的有：

| Intrinsic | 功能 |
|-----------|------|
| `_mpy()` | 16×16 位乘法 |
| `_mpy2()` | 两路 16×16 乘法（一条指令做两次） |
| `_dotp2()` | 两个 16 位数的点积 |
| `_add2()` | 两路 16 位并行加法 |
| `_sub2()` | 两路 16 位并行减法 |
| `_max2()`, `_min2()` | 两路 16 位并行求最大/最小 |
| `_amem4()`, `_amem8()` | 按 4/8 字节对齐加载内存 |
| `_ext()`, `_extu()` | 位域提取（有符号/无符号） |
| `_sadd()`, `_ssub()` | 饱和加减（防溢出） |

#### 10.4.2 一个例子

**朴素写法**（两路乘法串行）：

```c
short a[2], b[2];
int r1 = a[0] * b[0];
int r2 = a[1] * b[1];
```

**Intrinsics 写法**（两路乘法一条指令并行）：

```c
short a[2], b[2];
int ab_packed = _amem4(a);     // 把 a[0..1] 当作一个 32 位打包数加载
int cd_packed = _amem4(b);
long long result = _mpy2(ab_packed, cd_packed);   // 一条指令做两次乘法
int r1 = _loll(result);   // 低 32 位
int r2 = _hill(result);   // 高 32 位
```

第二种写法在 C66x 上**一个指令周期**就能完成两次乘法（借助 `.M1` 或 `.M2` 单元的 SIMD 能力）。第一种写法虽然编译器可能也能优化成一样的，但不保证——用 Intrinsics 能**确定性地**让你想要的指令出现。

#### 10.4.3 什么时候用 Intrinsics

不要一开始就用 Intrinsics。流程应该是：

1. 先用普通 C 写代码，确保正确
2. 开 `-O3` 编译，测性能
3. 如果性能不够，用 Profiler 找瓶颈
4. **只对瓶颈热点**尝试 Intrinsics
5. 改完再测，对比前后性能

Intrinsics 会让代码可读性变差，不到必要不要用。而且**某些算法完全不需要 Intrinsics**——比如本项目里的控制算法，核心是几十个乘加运算，编译器 `-O3` 优化已经能做得很好。Intrinsics 主要在图像处理、FFT、FIR 滤波这种**纯数据流**的场景收益大。

### 10.5 循环展开与软件流水

#### 10.5.1 循环展开（Loop Unrolling）

**循环展开**就是把一个循环的循环体**复制几份**，减少循环次数。比如：

```c
// 原始
for (int i = 0; i < 100; i++) {
    sum += a[i] * b[i];
}

// 手动展开 4 倍
for (int i = 0; i < 100; i += 4) {
    sum += a[i] * b[i];
    sum += a[i+1] * b[i+1];
    sum += a[i+2] * b[i+2];
    sum += a[i+3] * b[i+3];
}
```

**为什么展开能加速？**
- 减少循环开销（比较、跳转指令）
- 给编译器更多机会把独立操作调度到不同功能单元
- 便于 SIMD 打包

编译器 `-O3` 会自动做展开，一般不用手写。但有时候编译器展开的因子不合适，你可以用 `#pragma UNROLL(n)` 显式指定：

```c
#pragma UNROLL(4)
for (int i = 0; i < 100; i++) {
    sum += a[i] * b[i];
}
```

#### 10.5.2 软件流水（Software Pipelining）

**软件流水**是 DSP 编译器最复杂、最重要的优化技术之一。思想是：**在循环的多个迭代之间"错开"指令，让它们在流水线的不同阶段并行执行**。

举个例子，一个简单的循环每次迭代需要 4 步：
1. 加载 `a[i]`
2. 加载 `b[i]`
3. 计算 `a[i] * b[i]`
4. 累加到 sum

朴素执行这 4 步需要 4 个周期，100 次迭代要 400 个周期。

软件流水后的效果是：

```
周期 1:  load a[0]
周期 2:  load a[1]  |  load b[0]
周期 3:  load a[2]  |  load b[1]  |  mul a[0]*b[0]
周期 4:  load a[3]  |  load b[2]  |  mul a[1]*b[1]  |  add → sum
周期 5:  load a[4]  |  load b[3]  |  mul a[2]*b[2]  |  add → sum
...
```

"稳态"每个周期完成一次完整的乘加，100 次迭代只要大约 100 + 4 个周期。**接近 4 倍加速**。

**软件流水几乎全靠编译器自动做**。你能做的是**写对编译器友好的代码**：

- 避免循环体里的分支（if/else 打断流水）
- 循环次数在编译时尽量可知
- 避免循环迭代之间的数据依赖
- 加 `#pragma MUST_ITERATE(min, max, multiple)` 告诉编译器循环次数的范围

编译器会在编译报告里告诉你"这个循环流水化了，周期数多少"，看到流水化失败时再想办法优化源代码。

### 10.6 其他常见优化技巧

#### 10.6.1 用 `restrict` 关键字

告诉编译器两个指针**不会指向同一块内存**，编译器可以更激进地并行化：

```c
void process(float * restrict in, float * restrict out, int n) {
    for (int i = 0; i < n; i++) {
        out[i] = in[i] * 2.0f;
    }
}
```

没有 `restrict` 时，编译器必须假设 `in` 和 `out` 可能指向同一地址，就不敢并行；加了 `restrict` 就可以放心展开和流水化。

#### 10.6.2 避免浮点和定点混用

频繁的 `int ↔ float` 转换有额外开销。要么都用定点，要么都用浮点，不要一会儿转来转去。

#### 10.6.3 避免除法

除法是 DSP 里**最慢的运算**（单精度浮点除法在 C66x 上要几十个周期）。能用乘以倒数代替就用乘：

```c
// 慢
for (int i = 0; i < n; i++) {
    out[i] = in[i] / k;
}

// 快（如果 k 是常数或不常变）
float inv_k = 1.0f / k;
for (int i = 0; i < n; i++) {
    out[i] = in[i] * inv_k;
}
```

#### 10.6.4 调用 TI 优化库

TI 提供了几个高度优化的数学库：

- **DSPLIB**：FFT、FIR/IIR 滤波、矩阵运算、排序等
- **IMGLIB**：图像处理（卷积、边缘检测、形态学等）
- **MATHLIB**：`sin`、`cos`、`exp`、`log` 等数学函数的 DSP 优化版本

这些库的函数都是 TI 工程师用汇编精心优化的，性能极高。**别自己造轮子**——如果你的需求在这些库里有，直接调用就行。

### 10.7 性能优化的心态

最后讲一点非技术的东西：**心态**。

1. **先测量，后优化**。不测的优化都是瞎猜。
2. **找瓶颈**。优化不在瓶颈上的代码等于没优化。
3. **比对前后**。每次优化都要有"优化前 X 周期，优化后 Y 周期"的数据。
4. **保留原始版本**。优化可能引入 bug，要能随时回退。
5. **避免微观优化**。省一两个周期没意义，除非那一两个周期出现在每秒执行百万次的内循环里。
6. **可读性优先**。除非是极限热点，宁可慢一点也要可读。六个月后回来看代码的那个人是你自己。

### 10.8 第 10 章小结

1. **架构决策是最大的优化**，局部优化只应该针对实测瓶颈
2. **EDMA 双缓冲**让 EDMA 和 CPU 并行工作，能把吞吐率翻倍
3. **Cache 优化**的核心是空间局部性和时间局部性
4. **Intrinsics** 让你直接用 C66x 特殊指令，牺牲可读性换性能
5. **循环展开和软件流水**主要靠编译器自动做，写对编译器友好的代码是关键
6. **调用 TI 优化库**（DSPLIB、IMGLIB、MATHLIB）是第一选择，别自己造轮子
7. **优化流程**：测量 → 找瓶颈 → 局部优化 → 再测量 → 确认收益

### 11.3 代码逐段精读

本项目 `IsrMotionControl()` 是一个状态机，根据系统状态（`GetSystemCurState()`）走不同的分支。最典型、最完整的分支是 **`Default` 状态**——正常运行时走这个。看代码：

```c
case Default:
    IsrPrologue();                              // ① 中断前奏
    ReadSensorTopHalf();                        // ② 启动 EDMA 读传感器
    ReceiveControlCommand();                    // ③ 同时接收上位机指令
    CheckSensorDataTransfer(40);                // ④ 等 EDMA 完成
    ADDataProcess();                            // ⑤ AD 预处理
    RulerDataProcess();                         // ⑥ 光栅尺解算
    NotifyOtherCores();                         // ⑦ 触发 IPC 唤醒从核
    MultiCoreShareData->Phase[0] = Free;        // ⑧ 初始化栅栏标志
    MultiCoreShareData->Phase[1] = Free;
    MultiCoreShareData->Phase[2] = Free;
    MultiCoreShareData->Phase[3] = Free;
    MultiCoreShareData->Phase[4] = Free;
    MultiCoreShareData->Phase[5] = Free;
    MultiCoreShareData->Phase[6] = Free;
    MultiCoreShareData->Phase[7] = Free;
    MultiCoreShareData->Phase[8] = Free;
    IsrEpilogue(130);                           // ⑨ 中断收尾
    ParseControlCommand();                      // ⑩ 解析上位机指令
    SetallAxisRefreshFlag(0);                   // ⑪ 清刷新标志
    ClearGlobalVar();                           // ⑫ 清全局变量
    // ... 状态判断 ...
break;

// 周期末尾（在 switch 外面，所有状态都会执行）
VxDooBellSend();                                // ⑬ SRIO 通知 FPGA
```

下面逐行详细解读每一步在做什么，涉及哪些前面章节的概念。

#### 11.3.1 ① IsrPrologue() — 中断前奏

**做什么**：保存当前 CPU 上下文（寄存器、状态位），屏蔽更低优先级的中断，为中断服务函数的执行做准备。

**涉及的概念**：
- **中断服务函数（ISR）**：硬件中断触发时 CPU 自动跳到 ISR 的入口，ISR 必须保存现场才能在返回时恢复到中断前的状态
- **裸机 + 中断驱动模型**（第 9 章）：本项目没有操作系统做调度，一切都围绕这个 ISR 展开

#### 11.3.2 ② ReadSensorTopHalf() — 启动 EDMA3

**做什么**：配置 EDMA3 的 PaRAM Set（源地址 = FPGA 映射的传感器寄存器，目的地址 = CORE0 L2 的数据缓冲区），然后**启动** EDMA 传输。**注意"top half"的命名——这只是上半部分，只启动不等待**，函数立刻返回。

**涉及的概念**：
- **EDMA3**（第 5 章）：硬件 DMA，不占 CPU 就能搬数据
- **PaRAM Set**（第 5 章）：EDMA 的"搬运任务工单"（srcAddr/destAddr/aCnt/bCnt）
- **top half / bottom half 模式**：中断处理分两半，上半做最紧急的事（启动硬件），下半做后续处理。这是 Linux 内核里也常用的模式

#### 11.3.3 ③ ReceiveControlCommand() — 并行做别的事

**做什么**：检查上位机有没有通过以太网发来新指令，如果有就把它存到接收缓冲区。注意这一步**和 EDMA 搬运是并行的**——EDMA 硬件在后台搬传感器，CPU 同时在处理以太网。

**涉及的概念**：
- **CPU 和 EDMA 并行工作**（第 10 章性能优化）：这是双缓冲思想的简化版，通过重叠硬件和 CPU 的工作来提高整体吞吐
- **LWIP 协议栈**跑在 CORE0：这是为什么所有对外通信都集中在 CORE0 的原因之一（第 2 章）

#### 11.3.4 ④ CheckSensorDataTransfer(40) — 等 EDMA 完成

**做什么**：轮询 EDMA 完成标志（或等待中断）。参数 `40` 可能是超时值或重试次数。如果 EDMA 搬完了就继续；如果迟迟不完成（超时），说明传感器接口有问题，可能会记录一个错误。

**涉及的概念**：
- **轮询 vs 中断**：这里用轮询而不是等中断，是因为 EDMA 搬运时间很短（几十微秒），上下文切换的代价比轮询还大
- **超时保护**：实时系统里所有"等待"操作都必须有超时，否则一次硬件故障会让整个系统卡死

#### 11.3.5 ⑤ ADDataProcess() — AD 预处理

**做什么**：对刚搬进来的 AD（Analog-to-Digital，模数转换）采样数据做预处理——校准零点、去除噪声、转换物理单位。这通常涉及滤波（比如简单的 IIR 低通滤波）。

**涉及的概念**：
- **DSP 的拿手好戏**（第 1 章）：滤波是 DSP 最经典的任务，几条 VLIW + SIMD 指令就能算一个 IIR 二阶节
- **定点运算**（第 3 章）：AD 原始数据是 16 位整数，通常用定点算预处理更快

#### 11.3.6 ⑥ RulerDataProcess() — 光栅尺解算

**做什么**：光栅尺是一种高精度位置传感器，输出通常是增量编码信号。这一步把原始的脉冲计数转换成实际的物理位置（毫米/微米），可能还要做细分插值（提高分辨率）、零点校准等。

**涉及的概念**：
- **光栅尺**：精密运动控制的核心传感器，精度可达纳米级
- **浮点运算**（第 3 章）：光栅尺数据涉及单位转换（脉冲数 × 每脉冲毫米数），用单精度浮点最合适

执行完这一步，**所有传感器数据都已经在 CORE0 的本地缓冲区里**，并且是预处理好的、可用的形式。下一步就是把这些数据发给从核。

#### 11.3.7 ⑦ NotifyOtherCores() — 唤醒从核

**做什么**：这个函数内部应该做了三件事：

1. 把传感器数据 **Writeback** 到 MSMC 共享内存（确保 CORE0 Cache 里的数据推到主存）
2. **KickUnlock()** 解锁 Kicker
3. **CSL_IPC_genGEMInterrupt()** 触发从核的 IPC 中断
4. **KickLock()** 锁回 Kicker

这是整个周期里多核协作的**关键一跳**。具体的唤醒代码在 `Initialize` 和 `SpecificTask` 分支里能明确看到：

```c
KickUnlock();
CSL_IPC_genGEMInterrupt(1, 0);  // 唤醒 CORE1
CSL_IPC_genGEMInterrupt(2, 0);  // 唤醒 CORE2
CSL_IPC_genGEMInterrupt(3, 0);  // 唤醒 CORE3
CSL_IPC_genGEMInterrupt(7, 0);  // 唤醒 CORE7
KickLock();
```

**涉及的概念**（全都是前三部分讲过的）：
- **MSMC 共享内存**（第 6 章）
- **Writeback Cache**（第 7 章）
- **IPC 中断**（第 6 章）
- **Kicker 防误写机制**（第 6 章）

**从这一刻起**，从核开始并行工作，而 CORE0 不等它们——CORE0 继续往下走做自己的事。

#### 11.3.8 ⑧ Phase[0..8] = Free — 初始化栅栏标志

**做什么**：把共享内存里的 `Phase[]` 数组全部置为 `Free`。这是告诉所有从核"**可以开始新一轮的工作**"。从核被 IPC 中断唤醒后会把自己负责的那个 Phase 位置改成 `Busy`，算完之后改回 `Free`。

**涉及的概念**：
- **栅栏同步**（第 7 章）：Phase[] 数组作为轻量级栅栏
- **时序隔离**（第 6 章）：这时候 CORE0 写 Phase[] 不会和从核冲突，因为从核还没开始写它们各自的那一项

**为什么要在唤醒从核之后才初始化 Phase？** 实际上顺序有讲究：在本项目里，从核的 ISR 会先读自己的 Phase 看有没有任务（或者 ISR 直接开始做，不读 Phase），然后再写回 Free。关键是**CORE0 不能在从核读/写 Phase 的同时也去动这个位置**。这里的代码顺序（先 IPC 后初始化 Phase）能工作，说明从核进入 ISR 到开始操作 Phase 之间有足够的延迟让 CORE0 把这 9 个赋值做完——这依赖于硬件中断延迟和指令时序的细节。在自己的项目里写类似代码时要小心**赋值顺序和同步时机**，可能的话先初始化 Phase 再发 IPC 更安全。

#### 11.3.9 ⑨ IsrEpilogue(130) — 中断收尾

**做什么**：恢复 CPU 上下文，准备返回。参数 `130` 很可能是**中断服务函数的最大预算周期数**（或一个标识）——用来做实时性监控，如果某个中断执行超过预算就报警。这是硬实时系统里常见的 WCET（最坏情况执行时间）监控。

**涉及的概念**：
- **硬实时监控**（第 8 章）：AMP 模式能保证时序确定性，但还需要运行时监控来发现异常

注意到一个有趣的细节：**IsrEpilogue 出现在 ParseControlCommand 之前**。也就是说"中断收尾"实际上不是真正的"ISR 最后一步"——这是一种编程惯例，表示"关键实时部分结束了，下面是可以被打断的普通处理"。这种做法在 TI 的 DSP 开发里不罕见。

#### 11.3.10 ⑩ ParseControlCommand() — 解析指令

**做什么**：解析前面 `ReceiveControlCommand()` 收到的上位机指令，更新系统的目标位置、运动模式、参数等。

这部分在从核并行计算的**同时**执行。也就是说：

```
CORE0:                  从核 CORE1/7:
  ...                     （IPC 中断唤醒）
  NotifyOtherCores() ─────→
  Phase[] = Free            (读共享内存)
  IsrEpilogue                (跑控制算法)
  ParseControlCommand        (写回结果)
  SetallAxisRefreshFlag      Phase[x] = Free
  ClearGlobalVar
  ...（可能的其他处理）
  CheckAllAxisFinish() ←───── (等所有从核完成)
  ...
```

这是**CPU 时间利用的黄金段**——CORE0 不在等从核，它在做下一轮需要的准备工作。运动控制系统里每一微秒都很宝贵，这种"重叠执行"的设计非常重要。

**涉及的概念**：
- **并行执行**：CORE0 和从核同时在做各自的事，这是 AMP 模式的核心价值
- **任务重叠**（第 10 章）：通过合理安排代码顺序让多个硬件资源同时工作

#### 11.3.11 ⑪⑫ 清刷新标志和全局变量

**做什么**：清理上一周期残留的状态，为下一周期做准备。这类"清理"函数是状态机编程的常见模式。

#### 11.3.12 ⑬ VxDooBellSend() — SRIO 发 FPGA

注意这个函数在 switch 外面，**所有状态分支结束后都会执行**。

**做什么**：通过 SRIO 的 DoorBell 机制把控制量发给 FPGA。具体调用链可能是 `VxDooBellSend()` → `SRIOWriteData()`（我们在第 5 章看过这个函数的代码）→ 底层 SRIO 硬件。

**涉及的概念**：
- **SRIO**（第 5 章）：DSP-FPGA 通信主干
- **DoorBell**（第 5 章）：轻量级通知机制
- **CORE0 是唯一和 FPGA 通信的核**（第 2 章、第 8 章）：外设隔离在 CORE0

**一个值得注意的细节**：`VxDooBellSend()` 在本代码里是**立即**发送当前周期末尾时共享内存里的控制量。如果这时候从核还没算完（`CheckAllAxisFinish` 还在等），会发生什么？

答案是：如果代码顺序是"**先等所有从核 → 再发 SRIO**"，那没问题；如果顺序是"**先发 SRIO → 后等从核**"，就可能发的是上一周期的陈旧数据。实际项目里这个顺序必须非常小心，具体怎么保证正确性要看完整的 `IsrMotionControl` 和 `VxDooBellSend` 的实现——可能 `VxDooBellSend` 内部带了栅栏检查，或者 SRIO 发送的其实是**上一周期**汇总好的控制量（这周期只启动发送，下周期再收集结果），形成一级流水化。运动控制里这种"**1 个周期的延迟**"是常见的设计折中。

### 11.4 输入整形：放在 CORE0 的代表性算法

本项目代码目录 `MCB_CORE0/control/` 下有一个 `inputshaping.c` 文件。**输入整形（Input Shaping）** 是运动控制里一种非常有特色的算法，值得单独讲一下，因为它能很好地说明"**为什么要把这个算法放在 CORE0 而不是从核**"。

#### 11.4.1 输入整形解决什么问题

想象一下：你有一个长的机械臂，末端挂着一个负载。你让电机快速加速然后停下——机械臂末端会**继续晃动**一段时间（残余振动），因为机械结构有弹性。普通的 PID 控制解决不了这个问题，因为 PID 是基于"**误差反馈**"的，而振动是"**执行结果**"的固有特性。

**输入整形**的思路是：**修改输入信号本身**，使得激发出来的振动模式相互抵消。具体做法是把输入信号（比如一个速度指令）**拆成若干个脉冲**，每个脉冲在特定时刻、特定幅度发出，使得系统振动的"正向激励"和"反向激励"相消。效果就像让机械臂走一个精心设计的"不对称轨迹"，到达目标位置时恰好没有振动。

**为什么叫 Input Shaping（输入整形）？** 因为它**塑造**（shape）了输入信号的形状。最简单的输入整形器叫 **ZV 整形器**（Zero Vibration Shaper），把一个阶跃指令变成两个间隔时间很短、幅度不同的阶跃，就能显著抑制残余振动。更复杂的有 **ZVD、EI、SI** 等整形器。

#### 11.4.2 为什么放在 CORE0

输入整形在本项目里放在 CORE0，这是有讲究的。原因：

1. **输入整形作用在"指令"上，而不是"反馈"上**。它需要的输入是上位机发来的原始运动指令（目标位置、目标速度），这些指令只有 CORE0 通过以太网收到。从核根本看不到上位机指令，自然也做不了输入整形。

2. **输入整形是"前处理"**。它的输出才是送给真正控制算法（位置环、速度环、PID）的输入。因此在数据流上，它应该在"感知阶段"之后、"派发阶段"之前完成——这恰好是 CORE0 的职责时段。

3. **输入整形的计算量很小**，通常只是几次卷积或简单的状态更新，放在 CORE0 一点都不累。从核的宝贵算力应该留给重量级的控制算法。

4. **输入整形需要一份全局参数**（系统的振动频率、阻尼比等）。把它放在 CORE0 集中管理，不用在核间同步参数。

所以 `MCB_CORE0/control/inputshaping.c` 的位置非常合理：它是 CORE0 "**感知 → 计算目标 → 发给从核**"这条链上的一个环节，位于 `RulerDataProcess()`（感知）和 `NotifyOtherCores()`（派发）之间的某个位置（可能封装在 `ParseControlCommand` 或其他函数里，也可能是独立调用的）。

#### 11.4.3 输入整形的典型输出

输入整形的输出会写进共享内存的"**参考轨迹区**"，从核读这个区域得到"**本周期每个轴的目标位置/速度**"，然后算位置环/速度环生成控制量。数据流大概是：

```
原始上位机指令 (CORE0 收到)
     ↓
InputShaping() (CORE0 里做整形)
     ↓
整形后的参考轨迹 (写入共享内存)
     ↓
从核读参考轨迹
     ↓
从核跑位置环/速度环 (计算误差、跑 PID、前馈补偿)
     ↓
控制量 (写回共享内存)
     ↓
CORE0 汇总
     ↓
SRIO → FPGA → 电流环 → PWM → 电机
```

你会看到——**前处理（CORE0）→ 主计算（从核）→ 后处理（CORE0）**的三段式结构，是带有视觉和复杂算法的运动控制系统的常见范式。

### 11.5 从核一侧的代码长什么样

前面一直在讲 CORE0 的 `IsrMotionControl`，从核的代码我们还没正面看过。这里大致描述一下从核的典型结构——

从核的 `main()` 函数大致是这样：

```c
int main(void) {
    // 初始化本核的 L1/L2 Cache 配置
    InitCache();

    // 初始化 IPC 中断接收：注册 ISR 到 IPC 中断号
    RegisterIPCInterrupt(CORE1_IPC_HANDLER);

    // 使能中断
    EnableInterrupts();

    // 主循环：等 IPC 中断到来
    while (1) {
        // 什么也不做，等中断
        asm(" IDLE");    // 进入低功耗等待
    }
}

// IPC 中断服务函数
void CORE1_IPC_HANDLER(void) {
    // 1. 清 IPC 中断标志
    IPC_ClearSourceInfo(CORE1);

    // 2. Invalidate 要读的共享内存区域
    InvalidateCache(&MultiCoreShareData->sensor, sizeof(SensorData));
    InvalidateCache(&MultiCoreShareData->ref, sizeof(RefData));

    // 3. 标记本核为 Busy
    MultiCoreShareData->Phase[my_axis] = Busy;

    // 4. 读取传感器数据和参考指令
    SensorData s = MultiCoreShareData->sensor[my_axis];
    RefData   r = MultiCoreShareData->ref[my_axis];

    // 5. 跑控制算法（位置环、速度环、前馈、陷波滤波等）
    ControlOutput out = RunControlLaw(s, r);

    // 6. 写回控制量
    MultiCoreShareData->control[my_axis] = out;
    WritebackCache(&MultiCoreShareData->control[my_axis], sizeof(ControlOutput));

    // 7. 标记本核为 Free
    MultiCoreShareData->Phase[my_axis] = Free;
    WritebackCache(&MultiCoreShareData->Phase[my_axis], sizeof(char));

    // 8. 返回
}
```

**关键特征**：

- **极简的主循环**：就是 `while(1) IDLE`，什么都不做，等中断
- **所有工作都在 ISR 里**：从核没有状态机、没有任务调度，ISR 就是全部
- **严格的 Cache 操作模板**：Invalidate → 读 → 计算 → 写 → Writeback
- **栅栏协议**：Busy → ... → Free
- **完全不碰外设**：从核的代码里看不到以太网、SRIO、FPGA 这些字眼

这种极简结构是 AMP 编程的美学——**从核越简单越好**。所有复杂度都集中在 CORE0，从核只做数学。

### 11.6 时序分析：一次控制周期有多快

假设控制周期是 100 微秒，那一次 `IsrMotionControl` 必须在 100 微秒内完整走完。大致的时间分配（估算，具体数值取决于项目）：

| 阶段 | 时间 | 说明 |
|------|------|------|
| ① IsrPrologue | ~1 μs | 上下文保存 |
| ② EDMA 启动 | ~1 μs | 配置并启动 EDMA |
| ③ 接收以太网（并行） | 0 μs | 和 EDMA 重叠，不占额外时间 |
| ④ 等 EDMA 完成 | ~10 μs | 取决于数据量 |
| ⑤ AD 预处理 | ~5 μs | 滤波 + 单位转换 |
| ⑥ 光栅尺解算 | ~10 μs | 浮点运算 |
| ⑦ 唤醒从核（IPC + Writeback） | ~2 μs | |
| ⑧ 初始化 Phase | ~0.1 μs | 9 个赋值 |
| ⑨ 输入整形 | ~5 μs | 可能包含在 ParseControlCommand 里 |
| ⑩ 并行：从核算 vs CORE0 做 ParseControlCommand | ~30 μs | 这是算法重头戏，时间窗是共享的 |
| ⑪ 等所有从核完成（CheckAllAxisFinish） | ~1 μs | 如果已经算完，几乎不等 |
| ⑫ 读取控制量（Invalidate + 读） | ~2 μs | |
| ⑬ 汇总 + SRIO DoorBell 发送 | ~5 μs | |
| ⑭ IsrEpilogue | ~1 μs | 上下文恢复 |
| **总计** | **~70 μs** | **留 30 μs 的裕度** |

**留 30% 裕度**是运动控制项目的常规做法——万一偶尔某个周期稍慢，不会超时。如果你测下来某个周期占用 90 μs，接近 100 μs 的硬限，系统会非常脆弱，一点点扰动就可能错过周期。这种时候就要优化（回到第 10 章的内容）或者**降低控制频率**（100 μs → 200 μs）。

为了让你清晰看到"**前面学的每个概念在这个真实项目里都用到了**"，下面做一个对照表：

| 前面章节的概念 | 在 IsrMotionControl 里的体现 |
|---------------|--------------------------|
| 第 1 章 DSP 的实时性 | 整个函数必须在 100 μs 内完成 |
| 第 2 章 KeyStone 多核 | 用了 CORE0 + CORE1/7 的分工 |
| 第 3 章 VLIW/浮点 | ADDataProcess、RulerDataProcess 里的滤波和解算 |

| 第 5 章 SRIO DoorBell | VxDooBellSend 通知 FPGA 执行 |

**这张表就是这份文档的核心价值**：它展示了一个看起来普通的 `IsrMotionControl` 函数背后蕴含了多核 DSP 开发的几乎所有关键概念。理解了这个函数，你就理解了多核 DSP 实战开发的绝大部分。

### 11.8 第 11 章小结

1. **一次控制周期分为七个阶段**：中断触发 → 感知 → 派发 → 并行计算 → 汇总 → 输出 → 等待
2. **IsrMotionControl() 是整个项目的心脏**，由 Timer 硬件中断周期性触发
3. **CORE0 的职责**是感知 + 派发 + 汇总 + 通信，**从核的职责**是纯计算
4. **任务重叠**是性能的关键：EDMA 搬传感器时 CPU 收以太网指令，从核算控制律时 CPU 解析上位机命令
5. **输入整形**放在 CORE0 的原因：作用在指令上、是前处理、计算量小、需要全局参数
6. **从核代码极简**：`while(1) IDLE` + 一个 ISR，所有复杂度压在 CORE0
7. **时序预算**通常留 30% 裕度，以应对偶发的重负载
8. **前 10 章的每个概念都能在这个真实函数里找到对应**——这就是文档的收束点
