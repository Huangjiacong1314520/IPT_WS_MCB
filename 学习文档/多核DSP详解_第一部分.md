## 第 1 章 什么是 DSP，什么是多核 DSP

### 1.1 从一个问题开始：为什么我们需要 DSP

先设想一个场景：你要设计一台精密运动控制设备，比如一台半导体光刻机的工件台，或者一台高精度数控机床。这台设备上有十几个电机轴，每个轴都有光栅尺（一种高精度位置传感器）反馈位置。你要做的事情是：

1. **每隔很短的时间**（比如 100 微秒）读一次所有光栅尺
2. 根据当前位置和目标位置的偏差，**算出**每个电机下一步该输出多少电流
3. 把电流指令发给驱动器
4. 中间还要做各种滤波、前馈补偿、振动抑制等算法

> <span style="color:#FF0000;">即：多轴高频的 读数，计算【+算法】，发送指令</span>。算力要求高，实时性要求严格。

这里有两个关键点让普通单片机（MCU，Microcontroller Unit，微控制单元）很头疼：

- **算力要求高**：一次控制周期里要做几十次甚至上百次乘加运算（比如 FIR 滤波、矩阵乘法、FFT 频谱分析），如果用 STM32 这种通用 MCU，算不过来。
- **实时性要求严格**：控制周期不能抖动，必须**准时**完成。差 1 微秒都可能让电机抖动或者精度下降。

**DSP（Digital Signal Processor，数字信号处理器）** 就是为这种场景设计的专用处理器。它和普通 CPU 最大的区别是：

| 特性 | 通用 CPU/MCU | DSP |
|------|------------|-----|
| 强项 | 分支判断、逻辑控制、操作系统调度 | 大量的乘加运算、矩阵运算、信号滤波 |
| 指令特点 | 一条指令做一件事 | 一条指令可以做多件事（比如一次完成"乘 + 加 + 取下一个数"） |
| 内存结构 | 通常是冯·诺依曼（指令和数据共用一条总线） | 哈佛结构（指令和数据分开，可以并行取） |
| 浮点能力 | 通用 CPU 有，低端 MCU 通常没有 | 中高端 DSP 内置高性能浮点单元 |
| 典型应用 | 手机、电脑、服务器、PLC | 音频处理、雷达、通信基带、电机控制、医疗成像 |

> **建议搜索图**：「DSP 与 MCU 架构对比」，可以看到哈佛结构和冯·诺依曼结构的差异图示。

### 1.2 什么是"多核 DSP"，它和多核 CPU 有什么区别

最早的 DSP 都是单核的，比如经典的 TI TMS320C54x 系列（用于手机语音处理）。但到了 2000 年代后期，单个 DSP 核的频率提升遇到瓶颈（频率越高，功耗和散热越可怕），于是厂商开始往一颗芯片里塞多个 DSP 核，这就是**多核 DSP**的由来。

**但多核 DSP 和我们熟悉的多核 CPU（比如电脑里的 Intel i7）不一样**，区别主要在三点：

**区别一：调度方式**

- **多核 CPU**：操作系统（Windows、Linux）负责把进程、线程动态分配到各个核上跑，你写代码时通常不需要关心"这段代码跑在哪个核上"。
- **多核 DSP**：大多数情况下**没有操作系统**，或者只有一个非常轻量的实时内核。每个核跑什么代码、什么时候跑，**都由程序员手工指定**。每个核往往有自己独立的一份可执行文件（`.out`）。

<span style="color:#3399FF;">原来.out是可执行文件</span>

> * 在电脑里，你双击一个 .exe，操作系统把它加载到内存运行。
> * 在多核 DSP 里，通常每个核都有自己的 .out 文件。**核 0.out**：负责开机引导、初始化设备、分发数据。**核 1.out**：只管闷头算数学题。
> * 这就像给不同的专家发了不同的“工作手册”，大家各司其职，互不干扰，从而实现了**最高的执行效率**。

<span style="color:#FF0000;">问题：</span>为什么多核DSP没有操作系统？为什么不是也写一个操作系统而是要程序员手工指定每个核跑什么代码、什么时候跑？

> #### 操作系统调度过程中可能出现延迟：
>
> 像Windows或者Linux这种操作系统的调度器的逻辑：我看现在哪个核比较闲，我就把这个任务扔给他。这个是包含不确定性的，任务在切换核心、保存现场、重新加载缓存的过程中，会产生**不可预知的延迟**。
>
> 而DSP场景下：手工指定，任务A永远在核0，数据永远在核0的1L内存，耗时恒定为80微秒。
>
> 操作系统调度的时候，有可能缓存没命中，耗时变成了120微秒，这多出来的20微秒，就是控制时延抖动。对于我们这种高速、高加速度设备，可能导致晶圆报废。
>
> 这就导致了不可能用操作系统。

<span style="color:#FF0000;">问题：</span>为什么多核DSP不像CPU一样所有核心访问内存的速度一样，而是要分本地内存和远端内存？

> 物理上的限制：芯片内部信号传输是有延迟的，必然导致离核心远的内存访问速度慢，离核心近的内存访问速度快。如果让8个核心访问同一块内存的速度一样快，那各个核心距离其物理路径必须相等。随着核心数量的增加，相当于圈越来越大，大家都一样快=大家都一样慢。
>
> 总线竞争问题：如果所有核心都共享同一块内存，必然会遇到竞争问题，比如核0和核1同时要数据，到底先给谁，如果给了其中一个，另一个必然是要等待，这一等待就带来了抖动，带来了不确定性。
>
> DSP追求的是极致的响应速度，以及没有不确定性，最终选择了本地私人内存。
>
> 当然，这种本地内存也导致了，如果让操作系统自动调度，他把本来在核心0上跑的任务挪动到核心7，可是要算的数据都还在核0的本地内存里。这就导致核7必须跨遥远的总线去核0读取数据，就会慢很多。
>
> 因此，要手工指定，任务+对应的数据分配到哪个核心。
>
> PS：所以这就和通用CPU产生了区别：
> 通用CPU：电脑里各个核心没区别，地位平等，谁干活都可以，所以操作系统可以随便分配。
> 多核DSP：核心0专门负责和外界通讯（网口、串口），核1专门负责光栅计算，核7专门负责解耦解算控制算法等等。每个核的职责在硬件设计初期就定死了，就不需要操作系统来临时分配任务了。

<span style="color:#FF0000;">问题：</span>为什么CPU有Cache（缓存），有RAM（内存），但是DSP是 Scratchpad Memory（紧耦合内存 / L1 SRAM）

> * CPU 发现要用的数据不在最快的那层存储里，被迫去慢速内存里找，导致计算“卡顿”。
>   * 放下手头的笔（**CPU 挂起/停顿**）。
>   * 起身穿衣服、下楼、走到仓库（**发出内存访问请求**）。
>   * 在几百万份文件里找到《B 方案》（**内存寻址**）。
>   * 抱着文件走回办公室（**数据传输**）。
>   * 把文件放进抽屉，重新坐下继续写。
> * 不可预知性：是因为你不知道什么时候会卡顿，也不知道会卡多久。
>   * 当你去仓库（内存）取数时，如果仓库门口正好有很多搬运工在搬别的东西（总线占用），你还得排队。这就导致同样的“取文件”动作，有时候花 50 纳秒，有时候花 200 纳秒。
> * DSP 的优势：通过取消自动缓存，改为手动管理内存，彻底消灭了“没命中”的可能性，从而实现了像手术刀一样的精准控制。

<span style="color:#FF0000;">问题：</span>通用操作系统（Windows、Linux）和实时操作系统（VxWorks）的区别

> #### DSP 里的 RTOS 到底在干什么？
>
> 1. 核心功能一：优先级抢占（Hard Real-Time Scheduling）
>
>    这是 RTOS 的灵魂。	
>
>    * **场景**：你正在跑一个“UI 刷新”任务（优先级低），这时“电机位置偏差”信号来了（优先级极高）。
>    * **RTOS 的动作**：它会在纳秒级的时间内强行暂停 UI 任务，让电机任务接管。而且，这种切换的时间是**固定且已知**的。
>    * **意义**：保证最重要的事情永远在第一时间发生。
>
> 2. 核心功能二：静态配置（Static Configuration）
>
>    正如你之前看到的，这是 RTOS 与 Windows 的巨大区别。
>
>    * **Windows 做法**：你运行 QQ，系统动态给它分一块内存，位置随机。
>    * **RTOS 做法**：在程序写好还没运行之前，程序员写一个 .cfg 文件（在 TI 环境下）：“任务 A，优先级 15，放在 L1P 内存地址 0x00001000 开始的 2KB 空间，给它分配 1KB 的堆栈，不准挪动。”
>    * **意义**：运行期间**没有意外**。内存不会碎片化，不会因为找不到内存而报错，更不会因为内存访问变慢而导致计算超时。
>
> 3. 核心功能三：轻量级中断处理
>
>    在 DSP 里，中断（Interrupt）是家常便饭。
>
>    * RTOS 优化了中断的进出流程。它能保证：从硬件信号触发，到 CPU 开始跑你的中断代码，中间只经过几十个时钟周期。
>    * 它不会像 Windows 那样在中断处理时还要过一层厚厚的驱动架构包。
>
> 
>
> #### 为什么要用RTOS
>
> 你的 DSP 有 8 个核。
>
> * 如果你不用 RTOS：你需要自己写代码去管理这 8 个核怎么同步、怎么传数据、怎么防止大家同时抢一个外设。这叫“裸机（Bare-metal）开发”，虽然快，但代码写多了会乱成一团麻。
> * **如果你用了 RTOS（如 SYS/BIOS）**：
>   * 你只需定义好 8 个核的任务。
>   * RTOS 帮你提供**“旗语（Semaphore）”**：比如核 0 算完了，给核 1 发个信号，核 1 马上动起来。
>   * RTOS 帮你管理**“队列（Queue）”**：让数据在核与核之间像流水线一样整齐传输。
>
> **它不是来“管”你的，它是来帮你“规范化”管理时间、任务和硬件资源的。它牺牲了易用性（需要你手工配置），换取了工业级设备最渴求的——绝对的稳定和确定。**
>
> 
>
> 很多 DSP 工程师其实不喜欢叫它们“操作系统”，更愿意叫它们**“实时内核（Kernel）”**。
>
> | 维度         | 通用操作系统 (Windows/Linux)     | DSP 实时内核 (RTOS)                            |
> | ------------ | -------------------------------- | ---------------------------------------------- |
> | **存在感**   | 强，用户直接面对桌面和文件系统   | 弱，用户只通过 API 调用它，甚至感觉不到它存在  |
> | **内存隔离** | 严密。一个程序崩了，系统通常不崩 | 基本没有。为了速度，程序可以直接写坏系统的内存 |
> | **文件系统** | 必须有磁盘、文件夹               | 通常没有，只有内存地址和寄存器                 |
> | **交互性**   | 支持鼠标、键盘、多窗口           | 通常只有几个标志位（Flag）和数据流             |



**区别二：对实时性的追求**

- **多核 CPU**：追求**平均性能**。偶尔一次任务慢一点没关系，只要总体吞吐量高就行。
- **多核 DSP**：追求**确定性**。每个任务必须在**规定时间内**完成，不能抖动。运动控制里如果一个控制周期迟到 10 微秒，电机可能就会抖动甚至损坏。

**区别三：和外设的关系**

- **多核 CPU**：外设（硬盘、网卡、显卡）都通过操作系统的驱动访问，应用程序不直接碰硬件。
- **多核 DSP**：核直接读写外设寄存器，直接配置 DMA（Direct Memory Access，直接内存访问），直接处理中断。每个核可以"独占"某些外设。

一句话总结：**多核 DSP = 多个专门算数的核 + 手工分工 + 硬实时**。

### 1.3 从单核到多核的演进动因

为什么厂商要做多核 DSP？主要有四个原因：

1. **频率瓶颈**：单核频率提上去后，功耗以频率的平方甚至立方增长。与其把一个核做到 2GHz 发烫，不如做 8 个 1GHz 的核并行。

2. **数据级并行天然适合**：DSP 处理的数据很多都是**天然可并行**的——比如一张图像的不同区域可以同时处理、一个雷达的不同波束可以同时形成、多个电机轴可以同时计算控制律。这种"相互独立的任务"扔给多核简直是完美匹配。

3. **功耗效率更高**：同样的总算力，多核比单核省电。对于嵌入式设备（无人机、便携医疗仪器）非常重要。

4. **隔离性和可靠性**：一个核崩溃，另一个核还能继续跑。对于航空航天、工业控制等高可靠场景，多核可以做冗余。

### 1.4 为什么运动控制会用到多核 DSP

你可能会想：运动控制不就是 PID 闭环吗？随便一个 STM32 或者 TI 的 C2000 系列 MCU（比如 F28335）就能跑，为什么要动用多核 DSP 这种"大家伙"？

答案是：**简单的运动控制不需要**，但**高端运动控制需要**。具体来说：

- **轴数多**：普通场景 1-4 个轴，高端场景可能 10 个以上轴要**同步**控制。每个轴的位置环、速度环都是计算量。
- **算法复杂**：除了 PID，还有输入整形（Input Shaping，一种抑制机械残余振动的算法）、前馈补偿、迭代学习控制（ILC）、模型预测控制（MPC）等，这些算法的运算量比 PID 大几十倍甚至上百倍。
- **信号处理任务**：编码器原始信号需要滤波、光栅尺信号需要细分插值、振动信号需要 FFT 频谱分析找到振动频率——这些都是 DSP 的拿手好戏。
- **视觉融合**：带相机的运动控制系统需要同时处理图像数据，图像处理本身就是 DSP/GPU 的传统战场。
- **严格的实时性**：所有上面的任务都必须在一个控制周期内**准时完成**。

当这些需求叠加起来，单核 MCU 或者单核 DSP 就不够用了。多核 DSP 可以把不同任务分到不同核上并行处理，既保证算力又保证实时性。这就是为什么高端运动控制会选择像 TI C6678 这样的八核 DSP。

> **建议搜索图**：「运动控制系统框图」，看一下典型的"上位机 → 控制器 → 驱动器 → 电机 → 编码器"数据流。

## 第 2 章 认识 TI C6678

### 2.1 C6678 是什么：一张名片

**TI C6678** 的完整型号是 `TMS320C6678`，是德州仪器（Texas Instruments，简称 TI）于 2010 年发布的一颗**八核定点/浮点 DSP**。它属于 TI 的 **KeyStone 架构**第一代产品，官方定位是高性能信号处理芯片。

它的"名片"信息：

| 项目 | 参数 |
|------|------|
| 厂商 | Texas Instruments（TI，德州仪器） |
| 架构系列 | KeyStone I |
| 核心数 | 8 个 C66x 核 |
| 单核主频 | 最高 1.25 GHz（常见 1.0 GHz） |
| 单核算力 | 最高 40 GMAC/s（定点）或 20 GFLOP/s（单精度浮点） |
| 总算力 | 最高 320 GMAC/s 或 160 GFLOP/s |
| L1 Cache | 每核 32 KB L1P（程序）+ 32 KB L1D（数据） |
| L2 SRAM | 每核 512 KB |
| 共享内存 | 4 MB MSMC SRAM |
| 外部内存 | 支持 DDR3，最大 2 GB |
| 典型外设 | SRIO、PCIe、千兆以太网、EDMA3、EMIF、GPIO、I2C、SPI、UART |
| 制程 | 40 nm |
| 封装 | 841 引脚 FCBGA |
| 典型功耗 | 约 10 W |

**几个关键缩写先解释清楚**：

- **MAC**：Multiply-Accumulate，乘加运算。DSP 最常做的操作（比如 `y = y + a*b`）就是一次 MAC。**GMAC/s** 就是每秒能做多少十亿次乘加。
- **FLOP**：Floating-point Operation，浮点运算。**GFLOP/s** 就是每秒多少十亿次浮点运算。
- **SRAM**：Static Random Access Memory，静态随机存取存储器。速度快、不需要刷新，但密度低、贵。片上缓存都是 SRAM。
- **DDR3**：Double Data Rate 3，第三代双倍数据率同步动态随机存取存储器。就是我们电脑里常说的"内存条"那种 DRAM，速度比 SRAM 慢但容量大便宜。
- **MSMC**：Multicore Shared Memory Controller，多核共享内存控制器。这块 4MB 的 SRAM 是**所有 8 个核都能访问的公共区域**，也是本项目中核间通信的主要场所，后面会重点讲。
- **EDMA3**：Enhanced Direct Memory Access v3，第三代增强型直接内存访问。一个专门负责"搬数据"的硬件模块，不占用 CPU 就能把数据从 A 搬到 B。
- **SRIO**：Serial RapidIO，串行快速 IO。一种高速芯片间通信协议，常用于 DSP 和 FPGA 之间的高速数据通路。在本项目里，C6678 通过 SRIO 和 FPGA 通信。
- **FPGA**：Field-Programmable Gate Array，现场可编程门阵列。一种"逻辑电路可以自己配置"的芯片，常用于需要极低延迟、硬件并行的底层控制（比如编码器采集、PWM 输出、电流环）。

> **建议搜索图**：「TI C6678 功能框图」或「C6678 block diagram」，TI 官方有一张非常清晰的结构图，可以看到八个核、L2、MSMC、DDR3 控制器、各种外设的完整布局。

<span style="color:#FF0000;">问题：</span>为什么算力会比主频高？

> 主频：芯片内部时钟信号振荡的速度
>
> 算力：芯片每秒钟到底完成了多少次数学运算，在DSP中，算力通常分为两个指标：
>
> * GMACs每秒十亿次乘累加运算，如a×b+c；
>   * 单核 40 GMAC/s。
>     * **为什么主频才 1.25GHz，算力能到 40G？**
>     * 这是因为 C66x 是 **VLIW（超长指令字）** 架构。它在一个时钟周期内，可以同时启动多个硬件乘法器同时工作。
>     * **计算公式**：`1.25 GHz×321.25 GHz×32`（意味着每个周期能做 32 个定点乘法）。
> * GFLOPs每秒十亿次浮点运算，就是带小数点的运算带小数点的计算，如 3.1415×1.2343
>   * 单核 20 GFLOP/s。
>     * 通常浮点算力是定点的一半或更低，因为浮点运算单元（FPU）更占芯片面积，资源更宝贵。
>
> 

### 2.2 KeyStone 架构全景

**KeyStone**（基石）是 TI 对这一代多核 DSP/SoC 架构的命名。它的核心思想可以用一句话概括：**八个 C66x DSP 核 + 一张高速片上网络 TeraNet + 一个共享内存控制器 MSMC + 一堆共享外设**。

简单画一下逻辑结构：

```
+---------+  +---------+  +---------+  +---------+
| Core 0  |  | Core 1  |  | Core 2  |  | Core 3  |
| L1P/L1D |  | L1P/L1D |  | L1P/L1D |  | L1P/L1D |
|   L2    |  |   L2    |  |   L2    |  |   L2    |
+----+----+  +----+----+  +----+----+  +----+----+
     |            |            |            |
     +------------+------------+------------+
                  |
         +--------+--------+           <-- TeraNet 片上网络
         |                 |
    +----+----+      +-----+-----+
    |  MSMC   |      | EDMA3 /   |
    | 4MB SRAM|      | 外设控制器 |
    +----+----+      +-----+-----+
         |                 |
    +----+----+      +-----+-----+
    |  DDR3   |      | SRIO/GbE/ |
    | 外部内存 |      | SPI/I2C等 |
    +---------+      +-----------+

+---------+  +---------+  +---------+  +---------+
| Core 4  |  | Core 5  |  | Core 6  |  | Core 7  |
| L1P/L1D |  | L1P/L1D |  | L1P/L1D |  | L1P/L1D |
|   L2    |  |   L2    |  |   L2    |  |   L2    |
+---------+  +---------+  +---------+  +---------+
```

**TeraNet** 是 KeyStone 架构的片上互连网络，可以理解为"芯片内部的高速公路网"，所有核、共享内存、外设都挂在这张网上。它的好处是：任何核都可以访问任何共享资源，而且多个核可以**同时**访问不同资源而不冲突（多条车道并行）。这是 KeyStone 能高效支持多核的关键。

### 2.3 八个 C66x 核的关键参数

**C66x** 是 TI 的第 6 代 DSP 核，名字里的 "66" 是代号，"x" 表示有多种变体。C6678 上的八个核完全相同，每个核都具备：

- **VLIW 指令集**（Very Long Instruction Word，超长指令字）：一个指令包里可以打包 8 条小指令，让 8 个功能单元同时执行。这是 DSP 高性能的关键，第 3 章会详细讲。
- **8 个功能单元**：2 个乘法器 `.M`、2 个算术单元 `.L`、2 个分支/位操作 `.S`、2 个访存单元 `.D`。
- **定点 + 浮点双能力**：既能跑整数运算也能跑单精度、双精度浮点。
- **每核私有的 L1P/L1D/L2 缓存**：L1P 存指令、L1D 存数据、L2 可以灵活配置为纯 SRAM 或缓存。
- **每个核都有自己的独立时钟、独立中断控制器、独立的寄存器组**。

这八个核在硬件上**完全对等**，没有"主核"和"从核"的区别——"主从"是**软件层面的角色划分**，是程序员指定的，不是硬件固定的。在本项目里，CORE0 被人为定义为"主控核"，但本质上 CORE0 和 CORE1 的硬件能力一模一样。

### 2.4 为什么实际项目常常只用 3-4 个核，而不是 8 个

C6678 有八个核，但你会发现很多真实项目只启用其中几个。本项目就是典型例子：**只使用 CORE0、CORE1、CORE7 三个工程**（以及 CORE1/2/3/7 作为逻辑从核，详见下文说明）。

原因有几个：

1. **算力够用**：运动控制的计算量虽然大，但 8 个核往往过剩。跑满 3-4 个核就能满足性能要求，剩下的核空闲反而有好处——散热均匀，电源裕度大。

2. **物理分布考虑**：选择 CORE0 + CORE1 + CORE7 而不是 CORE0 + CORE1 + CORE2，是因为 CORE7 在物理布局上离 CORE0/1 比较远，**高负载的核在空间上分散开散热更好**。这是硬件工程师和软件架构师会一起考虑的细节。

3. **简化软件复杂度**：每多启用一个核，就多一份工程、多一份调试成本、多一组核间通信要维护。在够用的前提下，核越少越省心。

4. **功耗考虑**：空闲核可以关闭时钟甚至下电，降低整机功耗。

**本项目的实际核分工**（从代码推断）：

- **CORE0（主控核）**：负责以太网通信（LWIP 协议栈）、CAN、RS485、SPI Flash、NAND、与 FPGA 的 SRIO 通信、相机接口（CamLink）、传感器数据采集（通过 EDMA3）、光栅尺数据解算、控制指令分发、向从核发送 IPC 中断唤醒信号。是**唯一和外设打交道的核**。
- **CORE1（计算从核）**：只有基础驱动（EDMA3、GPIO、SPI、IPC、Cache、Control），不碰任何对外通信。纯粹做数学运算——跑控制算法、算控制量。
- **CORE7（计算从核）**：和 CORE1 类似，主要做并行计算。

**关于从核数量的说明**：在 `MCB_CORE0/control/interrupt.c` 的中断服务函数 `IsrMotionControl()` 中，可以看到 CORE0 触发 IPC 中断唤醒的是 **CORE1、CORE2、CORE3、CORE7 四个核**：

```c
KickUnlock();
CSL_IPC_genGEMInterrupt(1, 0);  // 唤醒 CORE1
CSL_IPC_genGEMInterrupt(2, 0);  // 唤醒 CORE2
CSL_IPC_genGEMInterrupt(3, 0);  // 唤醒 CORE3
CSL_IPC_genGEMInterrupt(7, 0);  // 唤醒 CORE7
KickLock();
```

这段代码表明系统**软件层面**支持最多四个从核并行。但实际烧录到硬件上运行的从核数量可能少于此数（例如仅 CORE1 和 CORE7）——代码保留对多核的支持是为了**未来扩展**，当前版本根据实际轴数和算力需求只启用必要的核。后文讲核间通信时会继续以"CORE0 主控 + CORE1/7 从核"为主要场景讲解。

### 2.5 C6678 在本运动控制系统中的角色

把视角拉高，看一下 C6678 在整个系统里处于什么位置。本项目的**整体系统架构**大致是这样的（从代码里的外设和通信接口反推）：

```
  [上位机 PC]
      |
      | 以太网 / CAN / RS485
      |
  +---v----------------------+
  |    TI C6678 DSP          |
  |  +---------------------+ |
  |  | CORE0 主控核        | |
  |  |  - 以太网 LWIP      | |
  |  |  - 指令解析          | |
  |  |  - 传感器采集 EDMA3 | |
  |  |  - 光栅尺解算        | |
  |  |  - 输入整形          | |
  |  |  - 参考曲线生成      | |
  |  |  - 汇总控制量        | |
  |  +----------+----------+ |
  |             | IPC+MSMC   |
  |  +----------v----------+ |
  |  | CORE1 / CORE7 从核  | |
  |  |  - 并行跑控制算法    | |
  |  |  - 算完回写共享内存  | |
  |  +---------------------+ |
  +-------------+------------+
                | SRIO (DoorBell)
                |
  +-------------v------------+
  |       FPGA 协处理器       |
  |  - 编码器采集            |
  |  - AD 采样（电流、温度） |
  |  - PWM 输出              |
  |  - 电流环（硬件级）       |
  +-------------+------------+
                | 模拟/数字信号
                |
  +-------------v------------+
  |    电机驱动器 + 电机      |
  |    + 光栅尺 + 编码器      |
  +--------------------------+
```

**C6678 的角色可以总结为**："上承上位机、下接 FPGA 的中间大脑"：

- **上**：通过以太网和上位机通信，接收运动指令、反馈状态
- **下**：通过 SRIO 和 FPGA 通信，接收传感器数据、发送控制量
- **中**：运行所有**算法层面**的代码——轨迹规划、输入整形、位置环、速度环、前馈补偿等

**为什么不把所有事情都扔给 FPGA？** 因为 FPGA 写起来是硬件描述语言（Verilog/VHDL），开发复杂的算法效率太低；FPGA 适合做固定的、高速的、硬件级别的事情（比如电流环、PWM），不适合做算法频繁迭代的浮点数学。

**为什么不把所有事情都扔给 C6678？** 因为 C6678 再快，对于纳秒级的电流环也有心无力。软件有中断延迟，时序没那么硬。所以把"软件擅长的算法"放 DSP、"硬件擅长的时序"放 FPGA，这叫"DSP + FPGA 异构协同"，是高端运动控制的经典架构。

> ## DSP 和 FPGA 的分工边界
>
> 从代码可以提炼出一条很清晰的原则：
>
> **DSP 做"想什么"，FPGA 做"怎么动"。**
>
> 具体展开：
>
> ---
>
> #### DSP 负责的事
>
> **1. 所有需要"思考"的计算**
>
> 位置规划、速度规划、PID 控制、插补算法——这些都在从核里跑。DSP 是浮点运算强、可编程性高的处理器，适合做复杂数学运算。
>
> **2. 传感器数据的采集和预处理**
>
> 光栅尺原始计数由 CORE0 通过 EDMA3 读进来，`RulerDataProcess()` 解算成物理位置，`ADDataProcess()` 把 AD 原始值换算成力/电流等物理量。这些转换逻辑复杂、需要定标参数，放 DSP 更容易修改和调试。
>
> **3. 上位机通信和任务调度**
>
> 以太网收发、指令解析、状态上报全在 CORE0。多核的调度（什么时候唤醒从核、检查哪些轴完成了）也在 CORE0 的 `IsrMotionControl()` 里。
>
> **4. 系统状态管理**
>
> `Silence → Activate → Default → Initialize → SpecificTask → Stop` 这个状态机在 DSP 里，决定系统整体处于什么阶段。
>
> ---
>
> #### FPGA 负责的事
>
> 从代码里能直接看到的证据只有两个接口：
>
> **1. 传感器原始数据的采集前端**
>
> `ReadSensorTopHalf()` 发起 EDMA3 传输，数据来源指向 FPGA 那边的地址空间。说明 AD 采样、光栅尺计数这些模拟/高速数字信号的接口电路在 FPGA 里，FPGA 负责把原始数字量整理好放在内存映射地址里，DSP 来取。
>
> **2. 执行端输出**
>
> 每个控制周期末尾 `VxDooBellSend()` 通过 SRIO 发 DoorBell 给 FPGA。这个"敲门"信号告诉 FPGA：本周期 DSP 算完了，可以把控制量执行出去。说明**把控制量转成实际驱动信号**（DAC 输出、PWM 生成、电流环执行）是 FPGA 的工作，不在 DSP 里。
>
> ---
>
> #### 为什么这样分？根本原因是两者的硬件特性不同
>
> | 特性   | DSP                                | FPGA                                   |
> | ------ | ---------------------------------- | -------------------------------------- |
> | 擅长   | 复杂浮点运算、分支逻辑、可编程算法 | 高速并行 IO、精确时序、模拟接口        |
> | 不擅长 | 纳秒级精确时序、多路高速 ADC 接口  | 复杂数学算法、灵活的控制逻辑           |
> | 适合做 | PID、插补、状态机、通信协议        | ADC 采样、编码器计数、PWM 输出、电流环 |
>
> 电流环通常在 FPGA 里跑还有另一个原因：它的控制频率往往比位置环高一个数量级（比如位置环 10kHz，电流环 100kHz），DSP 的中断调度开销在这个频率下会成问题，FPGA 的硬件逻辑没有这个限制。
>
> ---
>
> #### 对你作为控制算法工程师的实际意义
>
> 你改的东西——PID 参数、轨迹规划、速度限制——全部在 DSP 侧，全部在你能直接看到和修改的代码里。
>
> **你的管辖范围到哪里结束**：从核把控制量写进 `MultiCoreShareData->CtrlOutput[i]`，CORE0 汇总后通过 SRIO 发给 FPGA，**从 SRIO 发出去的那一刻起就出了你的管辖范围**。FPGA 拿到控制量之后怎么驱动电机，那是 FPGA 工程师的事，你看不到也不需要管。

>
> #### 核心比喻：脊髓反射 vs 大脑思考
>
> 在精密运动控制系统中，DSP 与 FPGA 的分工可以用人体的神经系统来类比：
>
> •   FPGA 是脊髓反射：不经过大脑，信号到来立刻响应，延迟由硬件电路的传播路径决定，与时钟边沿严格对齐。
>
> •   DSP 是大脑思考：接收脊髓汇报的数据，运行复杂算法，作出决策，再把指令下发给脊髓执行。
>
>  
>
> 这个分工不是随意的，而是由两种芯片各自的物理特性和硬件资源决定的。
>
> #### 一、FPGA 的领地：底层（脊髓）层
>
> ##### 1.1 具体任务
>
> •   传感器信号采集：对接光栅尺（EnDat/BiSS 协议）、A/B 相正交编码器，实时计数。
>
> •   电流环控制：电机控制最内层的环路，运行频率 50kHz～100kHz。
>
> •   PWM 脉宽调制：生成控制功率器件（IGBT/MOSFET）通断的六路互补 PWM，包含精确死区。
>
> •   硬限位安全监控：碰撞传感器触发时，硬件电路直接关断输出，不经过任何软件判断。
>
>  
>
> ##### 1.2 为什么这些任务必须由 FPGA 完成
>
> **①** **编码器计数：DSP 缺少专用硬件接口**
>
> 光栅尺输出 A/B 两路正交方波，分辨率 1nm 时电机运行可产生每秒数百万次边沿跳变，每次跳变间隔仅约 200 纳秒。
>
> C6678 DSP 片上没有 QEP（Quadrature Encoder Pulse）硬件模块。FPGA 每个引脚背后有独立触发器，时钟每跳一次，所有引脚状态同时被采样，计数器自动更新，无需 CPU 介入。如果用 DSP GPIO 中断来实现，多路编码器同时跳变时中断会排队串行处理，无法并行。
>
>  
>
> **② ADC** **采样触发：DSP 写寄存器与时钟边沿之间存在相位不确定性**
>
> 高精度 ADC 需要在严格固定的时刻触发采样。FPGA 的触发信号由硬件计数器产生，每次都在完全相同的时钟边沿输出，抖动在皮秒量级。
>
> 即使是确定性执行的裸机 DSP，"写 IO 寄存器" 这条指令的执行时刻与时钟边沿之间没有固定的相位关系，受流水线状态和总线仲裁影响。这不是随机调度带来的抖动，而是软件指令和硬件时钟之间固有的相位不确定性——即使执行时间完全固定，触发时刻仍然无法对齐到时钟边沿。
>
>  
>
> **③ PWM** **与电流环：DSP 控制周期粒度比电流环要求还粗**
>
> 电流环需要在每个 PWM 周期内完成"采样 → 计算 → 输出"的完整闭环。PWM 频率 20kHz 时，周期只有 50 微秒，且采样必须精确发生在 PWM 周期中点（开关噪声最小的时刻），时序要求误差小于 1 微秒。
>
> C6678 的控制周期在 100 微秒量级，本身就比电流环周期（50 微秒）更粗。此外 C6678 没有三相 PWM 发生器和硬件死区控制器，六路带死区的互补 PWM 无法用软件可靠实现。
>
>  
>
> **④** **并行性：16 轴同时采集，FPGA 无需轮询**
>
> 如果设备有 16 个轴，FPGA 可以在逻辑上直接例化 16 个独立的编码器计数模块，16 路信号在同一时钟周期同时被读入，互不干扰。DSP 如果用中断轮询，16 路只能串行处理，引入累积延迟。
>
> #### 二、DSP 的领地：核心（大脑）层
>
> ##### 2.1 具体任务
>
> •   轨迹规划：计算电机如何平滑地从 A 点运动到 B 点（三阶/五阶多项式、S 型曲线）。
>
> •   位置环与速度环控制：根据反馈位置/速度运行 PID 或 ADRC 算法，输出力/电流指令。
>
> •   复杂补偿算法：非线性摩擦力补偿、振动抑制、多轴解耦、热漂移补偿等。
>
> •   通信与指令管理：处理以太网上位机通信（LWIP 协议栈），解析运动指令，上报系统状态。
>
> •   多核调度：CORE0 负责传感器采集、任务分发；CORE1/2/3/7 并行计算各轴算法。
>
>  
>
> ##### 2.2 为什么这些任务必须由 DSP 完成
>
> **①** **复杂浮点运算：FPGA 开发成本极高**
>
> 运动控制涉及大量浮点运算、三角函数、矩阵乘法。
>
> •   在 DSP 里：一行 C 代码 y = sin(x) * k + b，硬件浮点单元瞬间完成。
>
> •   在 FPGA 里：需要调用 IP 核、处理流水线延迟、处理定点化精度损失，开发周期以周计。
>
>  
>
> **②** **算法迭代灵活：FPGA 修改代价过高**
>
> 控制算法在调试阶段需要频繁修改。
>
> •   在 DSP 里：修改两行 C 代码，重新编译下载只需数秒。
>
> •   在 FPGA 里：修改一点逻辑可能导致时序违例，重新综合布线可能需要半小时以上。
>
>  
>
> **③** **确定性执行：裸机 DSP 是实时控制的合适选择**
>
> 该系统没有操作系统，采用手工绑核——任务 A 永远在 CORE0，数据永远在 CORE0 的 L1 内存，执行时间恒定。这正是精密控制所需要的：确定性而非最快速度。
>
> 操作系统的调度器会在核之间动态分配任务，产生缓存未命中、现场切换等不可预知延迟，对于高速高加速度设备可能导致晶圆报废，因此该系统的设计选择从根本上排除了操作系统。
>
> #### 三、DSP 与 FPGA 如何协同工作
>
> ##### 3.1 通信通道：SRIO（Serial RapidIO）
>
> 两者通过 SRIO 高速串行总线连接，有三种数据传输方式：
>
> •   NRead：DSP 主动向 FPGA 发读请求，FPGA 返回传感器数据（传感器方向）。
>
> •   NWrite：DSP 主动向 FPGA 写数据，把各轴控制量写入 FPGA 指定寄存器地址（控制量方向）。
>
> •   DoorBell：DSP 每个控制周期末尾发送的"周期完成"信号，FPGA 收到后才把控制量执行输出。
>
>  
>
> ##### 3.2 一个完整控制周期的协同时序
>
>  
>
> | **步骤** | **FPGA**  **动作**                      | **通信通道**                  | **DSP**  **动作**                                  |
> | -------- | --------------------------------------- | ----------------------------- | -------------------------------------------------- |
> | **①**    | 多路传感器数据缓存在内部寄存器          | SRIO NRead（DSP主动读）       | CORE0发起EDMA3请求，ReadSensorTopHalf()            |
> | **②**    | 等待，持续刷新传感器数据                | （无通信）                    | CORE0数据预处理；IPC唤醒CORE1/2/3/7；各核并行跑PID |
> | **③**    | 暂存控制量，等待DoorBell                | SRIO NWrite（DSP主动写）      | CORE0将各轴控制量写入FPGA指定地址                  |
> | **④**    | 收到DoorBell，执行输出：DAC/PWM驱动电机 | SRIO DoorBell（周期完成信号） | VxDooBellSend()——每周期末尾必然执行                |
>
>  
>
> ##### 3.3 DoorBell 机制的关键作用
>
> FPGA 不是一收到 NWrite 写入的控制量就立刻执行，而是等待 DoorBell 信号。这个设计保证了控制量的一致性：所有轴的新控制量都计算完毕（所有从核的 Phase[] 均为 Free）后，CORE0 才发出 DoorBell，FPGA 在同一时刻把所有轴的新值一起施加，不会出现 1 号轴用新值、2 号轴还在用旧值的情况。
>
> #### 四、为什么不「合二为一」
>
> ##### 4.1 为什么不全给 FPGA
>
> •   开发成本：用 Verilog 实现高性能浮点矩阵运算，开发周期以月计，且极易出错。
>
> •   资源消耗：浮点运算极耗 FPGA 的 LUT 和 DSP Slice，所有任务塞给 FPGA 需要价格数万元的超大型器件。
>
> •   迭代效率：每次算法调整都需要重新综合布线，开发周期远长于 DSP 重编译。
>
>  
>
> ##### 4.2 为什么不全给 DSP
>
> •   缺少专用硬件外设：C6678 没有 QEP 编码器接口、没有三相 PWM 发生器、没有硬件死区控制器，这些功能无法用软件完整替代。
>
> •   控制周期粒度不够细：DSP 控制周期约 100 微秒，电流环要求 50 微秒内完成完整闭环，粒度上就无法满足。
>
> •   采样时序精度不足：写寄存器指令与硬件时钟边沿之间存在固有相位不确定性，高精度 ADC 采样触发需要皮秒级对齐，软件指令无法达到。
>
> #### 五、总结对比
>
> | **特性**         | **FPGA**（脊髓层）                      | **DSP**（大脑层）                              |
> | ---------------- | --------------------------------------- | ---------------------------------------------- |
> | 主要任务         | 传感器采集、编码器计数、PWM生成、电流环 | 轨迹规划、位置环/速度环PID、误差补偿、指令解析 |
> | 处理模式         | 纯并行（多路信号同时处理）              | 串行+向量（按步骤高效计算）                    |
> | 开发语言         | Verilog / VHDL（描述电路）              | C / C++（描述算法）                            |
> | 时间尺度         | 纳秒级（时序由硬件电路决定）            | 微秒级（控制周期稳定确定）                     |
> | 数学强项         | 定点加减、逻辑移位、位运算              | 浮点运算、三角函数、矩阵运算                   |
> | 为何不能替代对方 | 写浮点算法开发成本极高；资源消耗大      | 缺少专用硬件外设；控制周期粒度比电流环要求还粗 |
>
>  
>
> DSP 与 FPGA 的组合不是简单的「快慢」之分，而是「硬件接口能力」与「算法计算能力」的互补。FPGA 做 DSP 物理上做不了的事（专用接口、纳秒级时序对齐），DSP 做 FPGA 开发代价极高的事（浮点算法、复杂逻辑）。两者通过 SRIO 的 NRead/NWrite/DoorBell 三种机制协同，在每个控制周期内完成一次完整的「采集 → 决策 → 执行」闭环。



<span style="color:#FF0000;">问题：</span>DSP和FPGA如何协同工作

> 我写的控制量最终怎么变成电机动作，传感器数据又怎么回来
>
> ---
>
> ## DSP 和 FPGA 如何协同：完整机制
>
> 整个协同关系围绕 **SRIO（Serial RapidIO）** 这条高速串行总线展开。你可以把 SRIO 理解成一条专用的"内部高速公路"，连接 DSP 和 FPGA，有三种不同的"车型"在上面跑。
>
> ---
>
> #### 第一种：DoorBell — 周期同步信号
>
> 这是最简单的，每个控制周期末尾：
>
> ```c
> VxDooBellSend();  // IsrMotionControl() 最后一行，必然执行
> ```
>
> DoorBell 包非常小，只携带一个16位的信息字，不传数据，就是一个"敲门"信号。含义是：**DSP 本周期所有计算已完成，请 FPGA 执行输出。**
>
> FPGA 收到这个信号之后，才会把 DSP 上一周期算好的控制量真正施加到电机上。这设计得很聪明——避免了 FPGA 在 DSP 还没算完的时候就用旧数据驱动电机。
>
> ---
>
> #### 第二种：NRead — DSP 主动从 FPGA 读数据（传感器方向）
>
> ```c
> SRIOReadData(FPGA_DEVICE_ID_8BIT, uRdAddr, 
>              SRIO_PACKET_TYPE_NRead, pLocalBuff, nLen);
> ```
>
> `NRead` 是 SRIO 的"读请求"包。DSP 发出一个读请求，指定 FPGA 内部的寄存器地址和长度，FPGA 收到后把对应数据发回来。
>
> 这就是传感器数据进来的通道。FPGA 持续采集 ADC 数据、光栅尺计数，缓存在自己内部的寄存器里。DSP 每个控制周期发 NRead 请求来取，取回来的原始数据再经过 `ADDataProcess()` 和 `RulerDataProcess()` 解算成物理量。
>
> ---
>
> #### 第三种：NWrite/StreamWrite — DSP 主动向 FPGA 写数据（控制量方向）
>
> ```c
> SRIOWriteData(FPGA_DEVICE_ID_8BIT, uWrAddr,
>               SRIO_PACKET_TYPE_NWRITE, pLocalBuffer, nLen);
> ```
>
> `NWrite` 是"写请求"包，不需要 FPGA 回应确认。DSP 把控制量（各轴的力/电流指令）写到 FPGA 指定的地址，FPGA 内部的逻辑电路读这个地址，然后把数字量转成 DAC 电压或 PWM 信号驱动电机。
>
> 被排除编译的 `srio_DA.c` 文件名（DA = Digital to Analog）正好印证了这个方向。
>
> ---
>
> #### 两个设备 ID 说明了什么
>
> ```c
> if (FPGA_DEVICE_ID_8BIT != cRemoteDevID_8Bit) {
>     cPort = 2;   // 换端口
>     cLSU  = 1;   // 换链路服务单元
> }
> ```
>
> 代码里有两种设备 ID 的判断分支——用不同的 SRIO 端口和 LSU 单元。说明系统里**至少有两个 SRIO 节点**，很可能是两块 FPGA 或一块 FPGA 的两个功能区（比如传感器 FPGA 和驱动 FPGA 分开）。
>
> ---
>
> #### 完整的一个控制周期，DSP↔FPGA 的交互时序
>
> ```
> ┌─────────────────────────────────────────────────────┐
> │  每个控制周期（Timer 中断触发）                        │
> │                                                     │
> │  ① DSP  →[SRIO NRead]→  FPGA                       │
> │     "给我最新的传感器数据"                            │
> │     FPGA 返回：ADC 原始值 + 光栅尺计数                │
> │                                                     │
> │  ② DSP 内部处理（与 SRIO 传输并行）                   │
> │     CORE0：数据预处理                                │
> │     CORE1~7：PID/插补计算                            │
> │                                                     │
> │  ③ DSP  →[SRIO NWrite]→  FPGA                      │
> │     "这是本周期各轴控制量"                            │
> │     FPGA 暂存，等待 DoorBell                         │
> │                                                     │
> │  ④ DSP  →[SRIO DoorBell]→  FPGA                    │
> │     "本周期完成，可以执行了"                          │
> │     FPGA：把控制量转成 DAC/PWM 驱动电机              │
> └─────────────────────────────────────────────────────┘
> ```
>
> ---
>
> #### 对你最重要的一句话
>
> **SRIO 是你代码的终点，也是 FPGA 代码的起点。**
>
> 你在从核里算出来的 `CtrlOutput[i]`，经过 CORE0 汇总，通过 `SRIOWriteData()` 写到 FPGA 的某个寄存器地址。从那个地址往后，是 FPGA 的 VHDL/Verilog 代码的世界，你看不到也不需要管。同样，`SRIOReadData()` 取回来的原始字节，在 `ADDataProcess()` 之前是 FPGA 世界的数据，之后才进入你的管辖范围。
>
> 你改算法、调参数，影响的是步骤②和③之间的数字。步骤①④是固定的基础设施。



### 2.6 第一部分小结

到这里你应该已经对多核 DSP 和 C6678 有了整体印象。关键要点回顾：

1. **DSP 是为乘加运算和实时信号处理设计的专用处理器**，和通用 CPU/MCU 区别很大。
2. **多核 DSP 的本质是"多个独立的计算核 + 手工分工 + 硬实时"**，不是多核 CPU 那种"操作系统自动调度"。
3. **C6678 是八核 KeyStone DSP**，每核带自己的 L1/L2 缓存，8 个核共享 4MB 的 MSMC SRAM 和 2GB DDR3。
4. **实际项目不一定用满 8 个核**，本项目采用 CORE0 主控 + CORE1/7 从核的模式（代码中保留了扩展到 4 个从核的能力）。
5. **C6678 在系统中的角色是"算法大脑"**，上通以太网到上位机，下通 SRIO 到 FPGA，形成 DSP + FPGA 异构协同架构。
