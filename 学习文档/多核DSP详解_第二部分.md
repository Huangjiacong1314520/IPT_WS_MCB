## 第 3 章 C66x 内核架构

### 3.1 先问一个问题：为什么 DSP 能算得这么快

在第 1 章里我们说过，DSP 一条指令可以做多件事。这句话听起来很模糊，这一章我们把它拆开讲清楚。

现代 DSP 快的原因可以归结为四个字：**并行执行**。但"并行"有很多种含义：

- **数据并行**：一条指令同时处理多个数据（SIMD）
- **指令并行**：同时执行多条不同的指令（VLIW 或超标量）
- **任务并行**：多个核同时跑不同任务（多核）
- **流水线并行**：一条指令分成多个阶段流水执行（pipeline）

C66x 核把这四种并行**全部用上了**。我们一个一个看。

### 3.2 VLIW：一条大指令，八件小事一起做

**VLIW** 的全称是 **Very Long Instruction Word**，中文叫**超长指令字**。名字听起来很唬人，但原理其实很朴素：**把多条小指令打包成一个"指令包"，一次性送进 CPU，让内部的多个功能单元同时执行。**

打个类比：普通 CPU 像一个厨师，一次只能做一道菜（取菜→切→炒→装盘→再取下一道）；VLIW 像一个拥有 8 个专用工位的大厨房，老板一声令下"同时炒 8 道菜"，8 个工位一起开火。

**C66x 核有 8 个功能单元**，分成两组（A 侧和 B 侧），每组 4 个：

```
A 侧：.L1   .S1   .M1   .D1
B 侧：.L2   .S2   .M2   .D2
```

每个功能单元的"专长"不同：

| 单元 | 名称 | 擅长的事情 |
|------|------|-----------|
| `.L` | Logic/ALU | 加减、比较、逻辑运算（与或非）、数据打包 |
| `.S` | Shift/Branch | 移位、分支跳转、位域操作 |
| `.M` | Multiplier | 乘法、乘加（MAC） |
| `.D` | Data/Load-Store | 从内存读数据、往内存写数据、地址计算 |

一个 VLIW **指令包**最多可以同时执行 **8 条指令**（A 侧 4 条 + B 侧 4 条），理论上一个时钟周期就完成 8 件事。在 1 GHz 主频下，这意味着每秒 80 亿次基本操作，这就是 DSP 算力的来源。

**举个实际例子：计算 `y = a*x + b*x`**

如果用普通 CPU，大概是这样的顺序（5 步）：

```
1. 从内存加载 a
2. 从内存加载 x
3. 计算 a*x
4. 计算 b*x
5. 把两个结果相加得到 y
```

如果用 C66x 的 VLIW，可以这样安排一个指令包（1 个周期）：

```
同一个周期里并行执行：
  .D1 加载 a
  .D2 加载 b
  .M1 计算 a*x       ← 假设 x 已经在寄存器里
  .M2 计算 b*x
  .L1 做加法 a*x + b*x
```

实际情况因为有流水线延迟，不会真的一周期完成这么多事，但这个例子说明了 VLIW 的核心思想：**把本来串行的操作尽量拆到同一个周期并行做**。

**谁来决定哪些指令可以打包？** 答案是**编译器**。这是 VLIW 和传统超标量 CPU（比如 Intel x86）最大的区别——后者是硬件动态决定，前者是编译器静态决定。所以 **DSP 编译器的优化能力对最终性能影响巨大**，第 10 章会讲到怎么引导编译器做更好的优化。

> **建议搜索图**：「C66x 功能单元框图」或「TMS320C66x functional units」，TI 官方有一张 A/B 两侧 8 个功能单元的结构图，看一眼就明白。

### 3.3 SIMD：一条指令同时处理多个数据

**SIMD** 的全称是 **Single Instruction Multiple Data**，中文叫**单指令多数据**。它和 VLIW 不冲突，是另一个维度的并行。

SIMD 的思想是：**一个寄存器里塞多个数据，一条指令同时对这些数据做同样的运算。**

C66x 核支持 128 位宽的 SIMD 操作。这意味着一个 128 位寄存器可以装：

- 16 个 8 位整数（比如图像像素），一条指令同时加 16 个数
- 8 个 16 位整数（比如 AD 采样值），一条指令同时乘 8 组
- 4 个 32 位整数或 4 个单精度浮点
- 2 个双精度浮点

**举个运动控制里的例子**：假设你有 8 个电机轴，每个轴的位置误差存在一个 16 位整数里。传统做法是用循环，每次算一个轴的 `P*error`：

```c
for (int i = 0; i < 8; i++) {
    output[i] = Kp * error[i];
}
```

这需要 8 次乘法，8 个周期（流水线满载的情况下）。

用 SIMD 的做法是：把 8 个 `error` 打包进一个 128 位寄存器，一条 SIMD 乘法指令同时算出 8 个 `Kp*error`，只要 1 条指令。速度提升接近 8 倍。

**SIMD 在代码里怎么用？** 一种办法是让编译器自动识别循环做**自动向量化**（auto-vectorization），另一种是手写**内在函数**（intrinsics，第 10 章细讲）。直接写汇编也行，但很少有人这么做了。

**VLIW 和 SIMD 结合起来**：VLIW 让 8 个功能单元并行执行 8 条指令，SIMD 让每条指令同时处理多个数据。两者叠加，理论峰值算力非常可观——这就是为什么 C66x 单核能到 40 GMAC/s。

### 3.4 定点与浮点：为什么 C66x 两个都要

早期 DSP 只支持定点运算（整数 + 小数用固定位数表示），因为浮点硬件贵、占面积、耗电。但浮点运算对程序员友好得多——不用手工管理缩放因子、不用担心溢出、算法移植自 MATLAB 也方便。

**C66x 是"定点+浮点"混合核**，两种都支持：

- **定点运算**：速度最快，能效最高。适合对性能极致要求的场景（通信基带、雷达）。
- **单精度浮点**（float，32 位）：速度次之，精度对大多数控制算法够用。运动控制里的主流选择。
- **双精度浮点**（double，64 位）：速度最慢，精度最高。某些高精度算法（比如长时间累积误差的积分）需要。

**运动控制里的典型选择**：位置环、速度环这些控制律用单精度浮点；涉及绝对位置的累积运算用双精度；底层的 AD 数据、编码器原始值用定点 16 位。这是一个权衡**精度、速度、代码可读性**的活。

### 3.5 从运动控制举一个具体例子：矩阵乘法

控制算法里经常要做矩阵乘法。比如一个状态空间模型 `x_next = A*x + B*u`，如果 A 是 6×6 矩阵，u 是 6 维向量，那 `A*x` 就是一个 6×6 矩阵乘 6×1 向量。

按朴素写法，这需要 36 次乘法 + 30 次加法。

C66x 用 VLIW + SIMD 做这件事大概是这样的思路：

1. `.D1` 和 `.D2` 同时从内存加载 A 的两行和 x 的两个元素（一周期加载 4 个数）
2. `.M1` 和 `.M2` 同时做两路乘法
3. `.L1` 和 `.L2` 同时做两路加法累积
4. SIMD 把单精度浮点打包成一次 2 路或 4 路并行
5. 这一切打包成一个 VLIW 指令包，一个周期完成多步

实际实现由编译器和 DSP 优化库（比如 TI 的 DSPLIB）完成，程序员只需要调用 `DSPF_sp_mat_mul()` 这样的函数。效果上，原本需要几百个周期的矩阵乘法可能被压缩到几十个周期。

### 3.6 流水线：再榨一层性能

**流水线**（pipeline）是 CPU 常见的技术，DSP 也用。思想是：把一条指令分成若干个阶段（取指→译码→执行→写回），每个阶段由不同的硬件单元负责，这样同一时刻可以有多条指令处于不同的阶段，像工厂流水线一样。

**C66x 的流水线有 11 级**（取指 4 级 + 译码 2 级 + 执行 5 级）。这意味着"指令从开始到完成"需要 11 个周期，但只要流水线不中断，**平均每个周期仍然能完成一条指令**（甚至借助 VLIW 完成 8 条）。

流水线最怕的是**中断**——分支跳转（if/else、循环跳回）、Cache 缺失、数据依赖都会让流水线"打嗝"。所以第 10 章讲优化时，会专门讲"怎么写代码让编译器更好地做软件流水（software pipelining）"，这是 DSP 优化里很有技巧的部分。

### 3.7 C66x 内核架构小结

第 3 章的要点：

1. **VLIW**：一个指令包 8 条指令并行，对应 8 个功能单元（A/B 两侧各 4 个）
2. **SIMD**：一条指令同时处理多个数据，128 位宽
3. **定点 + 浮点混合**：可以根据需要选精度
4. **11 级流水线**：保持流水线不中断是高性能的关键

**这些特性对程序员意味着什么？** 意味着你**写普通 C 代码也能跑**，但要想发挥 C66x 的真正算力，必须：

- 让编译器开高优化等级（`-O3`）
- 合理组织数据结构让 SIMD 能并行加载
- 避免不必要的分支和数据依赖
- 必要时调用 TI 的优化库（DSPLIB、IMGLIB）或手写内在函数

第 10 章会展开讲这些优化手法。

## 第 4 章 存储层次：数据放哪里决定一切

### 4.1 一个关键事实：CPU 再快，也等得起内存

**现代 CPU/DSP 的瓶颈早就不是计算速度，而是内存速度。** C66x 核主频 1 GHz，一个时钟周期 1 纳秒。但如果它要从 DDR3 内存里读一个数据，往返可能要 100+ 纳秒——**等一个数据的时间够它算完 100 次乘法**。

这个矛盾叫"**存储墙**"（Memory Wall）。解决办法是搞一个分层的存储系统：最靠近 CPU 的存储最快但最小，最远的存储最慢但最大。数据按"冷热"放到不同层级。

### 4.2 C6678 的五层存储金字塔

C6678 的存储层次从近到远依次是：

```
       +----------+   ← 最快、最小、最贵
       |   寄存器  |     每核 32 个 32 位通用寄存器
       +----+-----+
            |
       +----v-----+
       |  L1P Cache |   每核 32 KB，纯指令
       |  L1D Cache |   每核 32 KB，纯数据，1 周期访问
       +----+-----+
            |
       +----v-----+
       |    L2    |   每核 512 KB，可配置为 Cache 或 SRAM
       +----+-----+
            |
       +----v-----+
       |   MSMC    |   全芯片共享，4 MB，~20 周期访问
       |  Shared   |
       +----+-----+
            |
       +----v-----+
       |   DDR3    |   外部内存，最大 2 GB，~100+ 周期访问
       +----------+   ← 最慢、最大、最便宜
```

**各层详细参数**：

| 层级 | 容量 | 延迟（近似） | 带宽 | 归属 |
|------|------|-------------|------|------|
| 寄存器 | 32×32 bit | 0 周期 | 不限 | 每核私有 |
| L1P | 32 KB | 1 周期 | 极高 | 每核私有 |
| L1D | 32 KB | 1 周期 | 极高 | 每核私有 |
| L2 | 512 KB | 4-8 周期 | 高 | 每核私有 |
| MSMC | 4 MB | ~20 周期 | 较高 | 所有核共享 |
| DDR3 | 2 GB | 100+ 周期 | 中等 | 所有核共享 |

**几个关键概念先解释**：

- **寄存器**：CPU 内部最小的存储单元，指令直接操作寄存器。C66x 每核有两组通用寄存器（A 组和 B 组），每组 32 个，每个 32 位。
- **Cache（高速缓存）**：用来"缓存"主存里频繁访问的数据的小块快速存储。当 CPU 读数据时，先查 Cache，命中了就快速返回，没命中（Cache Miss）才去下一级。
- **SRAM vs Cache 的区别**：同样是快速存储，SRAM 是"程序员显式放数据的地方"（你写 `*p = 5` 就是写进去了），Cache 是"硬件自动管理的副本"（你不知道哪些数据正好在 Cache 里）。

### 4.3 L1P 和 L1D：最快的贴身小仓库

**L1P**（Level 1 Program，一级指令 Cache）：**32 KB**，专门放指令。CPU 取指令时先查这里。它**只能作为 Cache**，不能当 SRAM 用。

**L1D**（Level 1 Data，一级数据 Cache）：**32 KB**，专门放数据。可以配置为纯 Cache、纯 SRAM 或者混合（比如 16 KB Cache + 16 KB SRAM）。

**L1D 有一个特别重要的特性：单周期访问。** CPU 一个时钟周期就能从 L1D 读到数据。这意味着如果你的数据能放进 L1D，DSP 就能全速运转。

**L1P/L1D 是每核私有的**。CORE0 的 L1D 里的数据，CORE1 根本看不到——后面讲核间通信时这是一个关键点。

### 4.4 L2：每核 512 KB 的本地快速内存

**L2** 是每核的"本地主战场"。512 KB 听起来不大，但对于嵌入式代码来说已经相当宽裕——放下整个控制程序的代码、主要数据结构、栈、堆绰绰有余。

L2 的一个关键特性是**可配置**：你可以告诉 C6678 "L2 的前 256 KB 当 SRAM 用，后 256 KB 当 Cache 用"。运动控制这种对延迟敏感的场景通常把 L2 **全部配置为 SRAM**，这样所有关键数据都能稳定地在"一个周期附近"的延迟访问，不会出现 Cache Miss 导致的抖动。

**本项目就是这么做的**。看一下 `MCB_CORE1/C6678_unified.cmd` 这个链接脚本：

```cmd
MEMORY
{
    LOCAL_L2_SRAM:  o = 0x00800000 l = 0x00080000   /* 512kB LOCAL L2/SRAM */
    LOCAL_L1P_SRAM: o = 0x00E00000 l = 0x00008000   /* 32kB LOCAL L1P/SRAM */
    LOCAL_L1D_SRAM: o = 0x00F00000 l = 0x00008000   /* 32kB LOCAL L1D/SRAM */
    SHRAM:          o = 0x0C000000 l = 0x00400000   /* 4MB Multicore shared Memmory */

    EMIF16_CS2:     o = 0x70000000 l = 0x04000000   /* 64MB EMIF16 CS2 Data Memory */
    EMIF16_CS3:     o = 0x74000000 l = 0x04000000
    EMIF16_CS4:     o = 0x78000000 l = 0x04000000
    EMIF16_CS5:     o = 0x7C000000 l = 0x04000000

    DDR3:           o = 0x80000000 l = 0x80000000   /* 2GB CE0 and CE1 external DDR3 SDRAM */
}

SECTIONS
{
    .text          >  LOCAL_L2_SRAM    /* 代码段 → L2 */
    .stack         >  LOCAL_L2_SRAM    /* 栈 → L2 */
    .bss           >  LOCAL_L2_SRAM    /* 未初始化全局变量 → L2 */
    .cio           >  LOCAL_L2_SRAM
    .const         >  LOCAL_L2_SRAM    /* 常量 → L2 */
    .data          >  LOCAL_L2_SRAM    /* 已初始化全局变量 → L2 */
    ...
}
```

**这个链接脚本在说什么？** 它告诉链接器：

1. **MEMORY 段**定义了所有可用的内存区域和它们的**物理地址**：
   - `LOCAL_L2_SRAM` 从 `0x00800000` 开始，512 KB
   - `LOCAL_L1P_SRAM` 从 `0x00E00000` 开始，32 KB
   - `LOCAL_L1D_SRAM` 从 `0x00F00000` 开始，32 KB
   - `SHRAM`（就是 MSMC）从 `0x0C000000` 开始，4 MB —— **这是核间共享内存的基地址**
   - `DDR3` 从 `0x80000000` 开始，2 GB

2. **SECTIONS 段**告诉链接器每种代码/数据放到哪个内存区域。这里所有段（`.text`、`.stack`、`.bss`、`.data`、`.const`…）**全部放进 L2**。

这个配置透露了一个重要信息：**本项目把所有代码和主要数据都塞进每核的 L2 SRAM，完全不用 DDR3 做运行时存储**（DDR3 可能只用来做 SPI Flash 启动时的镜像中转或者存大块日志）。这样做的好处是：

- **访问延迟稳定**：L2 SRAM 的延迟是固定的 4-8 周期，没有 Cache Miss 带来的抖动
- **没有 Cache 一致性问题**（在这一层）：因为不用 Cache 管理 L2，数据就是物理存在 L2 里
- **实时性强**：控制周期的每次计算时间可预测

**代价是**：每个核的所有代码必须能塞进 512 KB。这对代码规模是一个硬性约束，所以运动控制项目里很少用巨大的第三方库（比如一个完整的 JSON 解析器都嫌大）。

### 4.5 MSMC：多核的"公共广场"

**MSMC** 的全称是 **Multicore Shared Memory Controller**，中文叫**多核共享内存控制器**。它管理着 **4 MB** 的 SRAM，地址从 `0x0C000000` 开始，**所有 8 个核都能直接读写**。

MSMC 在本项目里的角色极其关键：**它是核间通信的主要场所**。CORE0 把传感器数据预处理完之后，写进 MSMC；CORE1/CORE7 再从 MSMC 读出来做计算；算完的控制量写回 MSMC；CORE0 再读出来发给 FPGA。整个控制周期的数据流经 MSMC 就像经过一个"公共广场"。

**MSMC 的访问延迟比 L2 慢**（约 20 周期），但比 DDR3 快得多，而且**对所有核的延迟是对称的**——CORE0 访问和 CORE7 访问用同样的时间。这个对称性对多核同步非常重要。

**但 MSMC 有一个大坑：Cache 一致性问题。** 因为每个核的 L1D/L2 都可以缓存 MSMC 里的数据，导致同一份数据在不同核的 Cache 里可能有不同副本。这个问题第 7 章会专门讲。

### 4.6 DDR3：最大的后备仓库

**DDR3** 是外部内存，本项目配置了 **2 GB**。它和主内存（比如电脑内存条）是同一类东西——容量大、便宜，但延迟高、带宽相对低。

DDR3 在运动控制里的典型用途：

- **启动镜像**：系统上电时从 SPI Flash 里读出程序镜像，先放到 DDR3，再分发到各核的 L2
- **大块日志/历史数据**：比如几十 MB 的轨迹记录、历史误差曲线
- **图像缓存**：如果系统接了相机，原始图像数据放 DDR3
- **不常访问的大型查找表**

**运行时热点数据一般不放 DDR3**，因为延迟太高会让控制周期抖动。这就是为什么本项目的链接脚本里大部分段都放 L2 而不是 DDR3。

### 4.7 数据应该放哪里：一个实用决策指南

给你一张"**数据应该放哪一层**"的速查表，以运动控制项目为例：

| 数据类型 | 放哪里 | 为什么 |
|---------|-------|--------|
| 代码段（.text） | L2 SRAM | 指令取用快，不抖动 |
| 栈、局部变量 | L2 SRAM | 频繁访问 |
| 每核私有的全局变量 | L2 SRAM | 每核独立，无共享问题 |
| 核间共享的数据结构 | MSMC（SHRAM） | 必须多核可见 |
| 核间共享且对延迟敏感 | MSMC + 严格 Cache 管理 | 靠手动刷新 Cache 保证一致 |
| 启动镜像、固件备份 | DDR3 | 只在启动时访问 |
| 相机图像帧缓冲 | DDR3 | 数据量大，L2 放不下 |
| 大型查找表（MB 级） | DDR3 或 L2（取决于访问频率） | 根据热度决定 |
| 常量表（几 KB） | L2 SRAM（.const 段） | 小而频繁访问 |

**一个基本原则**：**越热的数据放越靠上**。热度 = 访问频率 × 对延迟的敏感度。

### 4.8 第 4 章小结

1. **C6678 有 5 层存储**：寄存器 → L1P/L1D → L2 → MSMC → DDR3，越往下越大越慢
2. **每核的 L1 和 L2 是私有的**，其他核看不到，所以私有数据放 L2 最高效
3. **MSMC 是核间共享的 4 MB SRAM**，所有核都能访问，是核间通信主战场，但要小心 Cache 一致性
4. **DDR3 大但慢**，不建议放运行时热点数据
5. **运动控制典型做法**：L2 配为纯 SRAM，全部代码和数据放 L2，共享数据放 MSMC，DDR3 用来存大块非实时数据
6. **链接脚本（.cmd 文件）决定数据放哪里**，是控制存储分配的唯一入口

## 第 5 章 片上互连与关键外设

### 5.1 TeraNet：芯片内部的高速公路

前面说过 C6678 有 8 个核 + L2 + MSMC + DDR3 控制器 + 一堆外设。这些模块之间总得有个东西连起来，这个东西就是 **TeraNet**。

**TeraNet** 是 TI 在 KeyStone 架构里使用的**片上互连网络**（Network-on-Chip，NoC），可以理解为芯片内部的"高速公路交换网"。它的特点是：

- **多通道并行**：多个核可以**同时**访问不同的目标，互不阻塞
- **高带宽**：每条通道上百 GB/s
- **主从式**：核是主设备（Master，发起访问），存储和外设是从设备（Slave）

**为什么需要 NoC？** 如果只有一条总线，8 个核想访问 MSMC 就得排队，性能立刻崩溃。NoC 相当于把总线拆成网格，支持并行。

**程序员需要直接操作 TeraNet 吗？** 不需要。它对软件是**透明的**——你只管读写地址，硬件自动路由。你唯一能感知到的是"为什么 CORE0 和 CORE7 同时访问 MSMC 不会冲突"——答案就是 TeraNet 在背后做了并行路由。

> **建议搜索图**：「KeyStone TeraNet block diagram」，TI 官方文档里有 TeraNet 的多级交换结构图。

### 5.2 EDMA3：让 CPU 从搬数据里解放出来

**EDMA3** 的全称是 **Enhanced Direct Memory Access version 3**，中文叫**增强型直接内存访问（第三代）**。这是 C6678 上最重要的外设之一，重要到几乎每个 DSP 项目都绕不开它。

**它解决什么问题？** 解决 **CPU 搬数据浪费算力** 的问题。

设想一个场景：FPGA 有一批传感器数据要给 DSP，DSP 需要把这批数据从 FPGA 映射的地址空间搬到自己的 L2 SRAM 做处理。如果用 CPU 一个字节一个字节地 `memcpy`，1 KB 数据要花几千个周期——这几千个周期里 CPU 什么别的事都干不了。

EDMA3 是一个**独立的硬件模块**，它的唯一职责就是**搬数据**。你告诉它"源地址、目的地址、搬多少字节、搬完了通知我"，然后 EDMA3 自己去搬，**CPU 在此期间可以去算别的**。搬完之后 EDMA3 触发一个中断，CPU 回头来处理完成事件。

**本项目里 EDMA3 的用途**（从 `IsrMotionControl()` 函数看）：

```c
case Default:
    IsrPrologue();
    ReadSensorTopHalf();       // ← 启动 EDMA3 读取传感器数据
    ReceiveControlCommand();   // ← CPU 此时去干别的事：接收上位机指令
    CheckSensorDataTransfer(40); // ← 回头等 EDMA3 完成
    ADDataProcess();           // ← 处理刚搬回来的数据
    RulerDataProcess();
    NotifyOtherCores();
    ...
```

注意 `ReadSensorTopHalf()` 和 `CheckSensorDataTransfer(40)` 之间夹着一个 `ReceiveControlCommand()`。这就是 EDMA3 的精髓：**启动搬运之后立刻让 CPU 去做别的有用的事**，直到搬运快完成了才回来等。如果没有 EDMA3，CPU 就只能站在那里死等。

#### 5.2.1 EDMA3 的核心概念：PaRAM Set

**PaRAM** 的全称是 **Parameter RAM**（参数 RAM）。每个 EDMA3 传输由一组 "**PaRAM Set**" 定义，相当于一张"搬运任务工单"。

本项目的 `MCB_CORE0/EDMA3/edma.h` 里定义了这个结构体：

```c
typedef struct EDMA3CCPaRAMEntry {
    unsigned int   opt;       // 选项字：中断使能、同步类型、优先级等
    unsigned int   srcAddr;   // 源地址（从哪儿搬）
    unsigned short aCnt;      // 每个"数组"的字节数（A 计数）
    unsigned short bCnt;      // 每"帧"里有几个数组（B 计数）
    unsigned int   destAddr;  // 目的地址（搬到哪儿）
    short  srcBIdx;           // 源数据跨 B 的步长
    short  destBIdx;          // 目标数据跨 B 的步长
    unsigned short linkAddr;  // 链接地址（可以级联多个 PaRAM Set）
    unsigned short bCntReload;// B 计数重加载值
    short  srcCIdx;           // 源数据跨 C 的步长
    short  destCIdx;          // 目标数据跨 C 的步长
    unsigned short cCnt;      // 有几"帧"（C 计数）
} EDMA3CCPaRAMEntry;
```

这个结构体看起来参数很多，但核心就四个数字：**srcAddr、destAddr、aCnt、bCnt**。其他的是高级用法。

**aCnt/bCnt/cCnt 是什么意思？** EDMA3 支持**三维传输**——想象你要从一个二维数组里按某种模式取数据：

- **aCnt**：一个"小块"有多少字节（A 维度）
- **bCnt**：一个"中块"包含多少个"小块"（B 维度）
- **cCnt**：一共有多少个"中块"（C 维度）
- **srcBIdx、srcCIdx**：每跳一个 B 或 C，源地址加多少
- **destBIdx、destCIdx**：每跳一个 B 或 C，目的地址加多少

**举个例子**：你有一张 640×480 的灰度图，想从中间截取一个 100×100 的子图。用普通 `memcpy` 要写一个循环，一次一行。用 EDMA3 只要配置一个 PaRAM Set：

```
aCnt = 100        // 每行 100 字节
bCnt = 100        // 100 行
srcBIdx = 640     // 源图每行跳 640 字节
destBIdx = 100    // 目标每行跳 100 字节
```

一次传输完成，CPU 全程没参与。

#### 5.2.2 EDMA3 的触发方式

EDMA3 可以被三种方式触发：

1. **事件触发（Event-triggered）**：外设产生事件自动启动。比如串口收到一个字节，或者 AD 转换完成。这是**最省 CPU 的方式**——CPU 完全不参与启动。
2. **手动触发（Manual）**：CPU 往寄存器写一个 1 启动。适合"数据好了，我现在想开始搬"的场景。
3. **QDMA 触发**：CPU 写某个 PaRAM 字段时自动启动。适合频繁小数据传输。

本项目里 `CheckSensorDataTransfer(40)` 这种"启动后回来检查"的用法是手动触发 + 轮询的组合。

#### 5.2.3 EDMA3 有多少通道

C6678 有 **3 个独立的 EDMA3 实例**（叫 TPCC0/1/2，Third-Party Channel Controller 的缩写），每个实例有 **64 个 DMA 通道 + 8 个 QDMA 通道**。加起来**近 200 个独立的传输通道**同时可用。这是为什么 DSP 能同时维持多路高速数据流的硬件基础——音频、相机、传感器、核间内存拷贝可以分别用不同通道并行搬运，互不干扰。

### 5.3 SRIO：和 FPGA 说话的高速通道

**SRIO** 的全称是 **Serial RapidIO**，中文叫**串行快速 IO**。它是一种**芯片间高速串行通信协议**，专为 DSP、FPGA、ASIC 之间的点对点通信设计。

**和 PCIe、以太网相比，SRIO 的特点**：

| 特性 | SRIO | PCIe | 以太网 |
|------|------|------|--------|
| 典型应用 | DSP/FPGA 互联 | CPU ↔ 外设卡 | 通用网络 |
| 延迟 | 极低（<1 微秒） | 低 | 高（几十微秒） |
| 拓扑 | 点对点或小交换网 | 树形 | 任意 |
| 协议栈 | 轻量 | 中等 | 重（TCP/IP） |
| 开发复杂度 | 中 | 高 | 高 |

SRIO 就是为"**两颗高性能芯片之间需要低延迟、高带宽的原始数据传输**"这个场景设计的。运动控制里 DSP 和 FPGA 的组合是典型应用。

#### 5.3.1 SRIO 的三种主要传输类型

1. **NREAD/NWRITE（直接 IO 读写）**：像访问内存一样读写对方的地址空间。你在 DSP 这边写 `*(volatile int*)0xXXXXXXXX = 123`，对方 FPGA 的某块内存就被写成 123。最灵活但需要双方事先约定地址映射。

2. **DoorBell（门铃）**：发一个**极小的消息**（16 位）过去，对方收到后触发一个中断。作用是"**通知**"——告诉对方"数据好了，快来处理"。

3. **Message（消息）**：发一段结构化的消息，类似发短信。适合不频繁的控制命令。

本项目主要用了 **NWRITE（直接写数据）+ DoorBell（通知）** 的组合。看 `MCB_CORE0/srio/drv_srio.c` 的 `SRIOWriteData()` 函数：

```c
SRIORWStatus  SRIOWriteData(unsigned char cRemoteDevID_8Bit,
                            unsigned int uWrAddr,
                            unsigned char cPktType,
                            unsigned char *pLocalBuffer,
                            int nLen)
{
    SRIOLSUConfig cfg;
    unsigned char cLSU = 0;
    unsigned char cType = 0;
    unsigned char cPort = 0;

    // 根据目标设备 ID 选择 SRIO 端口和 LSU 通道
    if (FPGA_DEVICE_ID_8BIT != cRemoteDevID_8Bit)
    {
        cPort = 2;
        cLSU = 1;
    }

    cType = SRIOPacketType(cPktType);

    // 构造 SRIO 传输配置
    SRIO_InitLsuCfg(&cfg, (Uint32)pLocalBuffer, uWrAddr, cType, nLen,
                    cRemoteDevID_8Bit, cPort);

    // 如果是 DoorBell 类型，带一个 16 位的通知码
    if(cType == SRIO_DoorBell)
    {
        cfg.DoorBell.Enable = TRUE;
        cfg.DoorBell.Info = pLocalBuffer[0]<<8 | pLocalBuffer[1];
    }
    else
    {
        cfg.DoorBell.Enable = FALSE;
        cfg.DoorBell.Info = 0xa5a5;
    }

    // 启动传输
    SRIODirectIOTransfer(cLSU, &cfg);

    // 等待传输完成
    ...
}
```

**代码在说什么？** 这个函数把本地缓冲区 `pLocalBuffer` 的 `nLen` 字节通过 SRIO 写到远端设备（由 `cRemoteDevID_8Bit` 标识）的地址 `uWrAddr` 处。如果是 DoorBell 类型还会附带一个 16 位通知码。`FPGA_DEVICE_ID_8BIT` 是项目里定义的 FPGA 的 SRIO 设备 ID，通过这个判断选择走哪个 SRIO 端口和哪个 LSU。

**LSU** 的全称是 **Load/Store Unit**（加载/存储单元），是 SRIO 硬件里负责发起传输的模块。C6678 的 SRIO 有多个 LSU，可以并行发起多路传输。

#### 5.3.2 DoorBell：最优雅的"通知"机制

看一下 `SRIOGetDoorbellStatus()` 这些函数：

```c
unsigned int SRIOGetDoorbellStatus()
{
    return SRIODoorBellInterruptGet(0);
}

void SRIOClearDoorbellIntcStatus(unsigned char Doorbell)
{
    SRIODoorBellInterruptClear(0, Doorbell);
}
```

**DoorBell 的工作流程是**：

1. FPGA 处理完一批采样数据，通过 SRIO 往 DSP 发一个 DoorBell，附带一个 16 位信息码（比如 `0x0001` 表示"编码器数据好了"）
2. DSP 这边的 SRIO 硬件收到 DoorBell，自动触发一个 CPU 中断
3. CPU 中断服务程序调用 `SRIOGetDoorbellStatus()` 读出信息码，知道是什么事件
4. 处理完后调用 `SRIOClearDoorbellIntcStatus()` 清除中断标志

**为什么要用 DoorBell 而不是直接轮询？** 轮询（polling）是 CPU 不断查"数据来了没"，浪费 CPU 时间；DoorBell 是硬件"**主动敲门**"，CPU 平时可以干别的事，被敲门的时候再过来处理。这是嵌入式系统里"**事件驱动**"设计的经典体现。

本项目里 `IsrMotionControl()` 末尾的 `VxDooBellSend()` 就是 DSP 反向用 DoorBell 通知 FPGA——"本周期控制量已经写入共享区域，FPGA 可以取了"。

### 5.4 其他常用外设一览

C6678 还有大量其他外设，本项目里用到的主要有：

| 外设 | 用途 | 本项目里的角色 |
|------|------|--------------|
| **GPIO** | 通用输入输出 | 状态指示灯、配置引脚、简单开关量 |
| **SPI** | 串行外设接口 | 启动时从 SPI Flash 读镜像 |
| **I2C** | 双线总线 | 读 EEPROM 里的配置信息 |
| **UART** | 异步串口 | 调试打印 |
| **GbE**（千兆以太网） | 以太网 MAC | 对上位机通信（LWIP 协议栈跑在 CORE0） |
| **EMIF16** | 外部内存接口 | 访问板上其他存储芯片 |
| **Timer** | 硬件定时器 | **产生控制周期的心跳中断**（整个系统的节拍器） |

**特别说一下 Timer**：本项目的控制周期是由一个硬件定时器产生的周期性中断驱动的，这个中断触发 `IsrMotionControl()`，每次触发就走一个完整的"读传感器→算→发指令"流程。**Timer 是整个系统的心跳**，它的稳定性直接决定控制的精度。

**本项目 CORE0 和 CORE1/7 的外设分工**：

- **CORE0 独占**：以太网、SRIO、FPGA 通信、相机接口（CamLink）、CAN、RS485、NAND、SPI Flash
- **CORE1/CORE7 只有**：GPIO、SPI、EDMA3、Cache、IPC、Control

为什么要这么分？**把对外通信全部压在 CORE0**，让 CORE1/7 完全隔离在外设之外，这样从核的代码可以简单到极致——没有协议栈、没有中断嵌套、没有外设锁，只要一个 IPC 中断响应函数 + 一段算法代码就够了。这是第 6 章会讲到的"**主从同步并行**"模式的硬件基础。

### 5.5 第 5 章小结

1. **TeraNet** 是 C6678 的片上互连网络，对程序员透明，但使多核并行访问成为可能
2. **EDMA3** 是"搬数据专业户"，让 CPU 可以一边算一边搬数据，是高性能 DSP 的标配
3. **EDMA3 PaRAM Set** 是一张三维传输工单，核心字段是 `srcAddr/destAddr/aCnt/bCnt`
4. **SRIO** 是 DSP-FPGA 通信的主干，低延迟、高带宽
5. **DoorBell** 是 SRIO 提供的轻量级通知机制，是事件驱动设计的典范
6. **本项目所有对外通信都集中在 CORE0**，从核只负责计算，这是主从并行架构的基础
